## Verification of the `&` escape (implementation summary §2)

The summary's claim was checked carefully because bash's `${//}` does *not*
universally treat `&` as a backreference — that depends on the `patsub_replacement`
shopt. Findings:

- `ralph-loop.sh` runs as a bash script under bash 5.2.21, where
  `patsub_replacement` is **on by default**. In that mode `&` in a `${//}`
  replacement expands to the matched text, so a naive substitution would
  corrupt `cd src && npm ...` (the `&&` would be replaced with the matched
  placeholder text). Reproduced: naive → `x a PP b y`; escaped → `x a && b y`.
- The dev's pre-escape (`${CHECKPOINT_CMD//&/\\&}`) is therefore **correct and
  necessary** under the actual runtime, and the real function emits a clean
  `cd src && npm run build && npm test --if-present`.

## Non-blocking observations (do not require a fix for this story)

1. **Bash-version portability of the `&` escape.** The escape is correct on
   bash 5.2+ (`patsub_replacement` on). On bash 4.x / 5.0 / 5.1, where that
   shopt does not exist and `&` is literal, the same escape would instead emit
   literal backslashes (`cd src \&\& npm ...`) into the prompt. The System Track
   CLAUDE.md only promises "Bash 4+", so this is a latent edge case. It has
   **zero current impact** — the function is standalone and not yet wired into
   `build_system_prompts()` (AC 10), and the demo environment is bash 5.2.21.
   Worth a hardening note when story 1.5 wires this in (e.g. a version-agnostic
   substitution or an explicit `shopt -s patsub_replacement` guard), but not a
   blocker now.
