## Summary

Replace the inline heredocs in `build_system_prompts()` with calls to `load_prompt_layers()`, then delete the now-unused heredoc definitions. This is the destructive refactor step that consolidates the modularization work from stories 1.1–1.3. ### Design Goal

The new `build_system_prompts()` must:
- Call `load_prompt_layers(role)` for each of the three roles (sm, dev, review)
- Assign results to `SYSTEM_PROMPT_SM`, `SYSTEM_PROMPT_DEV`, `SYSTEM_PROMPT_REVIEW`
- Preserve all logging behavior (same log lines, same byte-size output)
- Produce semantically equivalent output to the current version

The three layers (as loaded by `load_prompt_layers()`) will produce the same final content because story 1.1 extracted the heredocs verbatim into the prompt files. The order will differ (Layer 1 before Layer 2, per the PRD's design), but significant content is preserved.

## Technical Details & Gotchas

### Heredoc Order vs. Layer Order

The original `build_system_prompts()` constructs prompts in this order:
```
[Persona / Fallback]
---
[Common (Execution Context + Project Conventions)]
```

The new `load_prompt_layers()` composes them as:
```
[Layer 1: Execution Context]
---
[Layer 2: BMAD Persona / Fallback]
---
[Layer 3: Project Conventions + Role Overlay]
```

This is intentional reordering per the PRD (Layer 1 must come first so "do not HALT" override wins against persona instructions). **The semantic content is identical**, but ordering differs. The verification script accounts for this by checking that every *significant line* from the baseline appears *somewhere* in the layered output, not that they appear in the same position.

### `{{CHECKPOINT_CMD}}` Substitution

The `load_prompt_layers()` function automatically substitutes `{{CHECKPOINT_CMD}}` with the value of the `CHECKPOINT_CMD` environment variable. The old heredoc in `build_system_prompts()` did this substitution via bash's `${CHECKPOINT_CMD}` expansion inside the heredoc.

The new version delegates substitution to the loader, so there is no need to do anything special in `build_system_prompts()` — just call the function and use the result.

### Fallback Handling

The old code had inline fallback stubs for each role (e.g., "You are the BMAD Developer..."). These are now in `scripts/prompts/bmad-fallbacks/{sm,dev,review}.md`, loaded by `load_prompt_layers()`. If `AGENT_*_PERSONA` is empty, the loader reads the fallback file automatically.

**No change needed** — the loader handles this transparently.

### Logging Preservation

The old function logged byte sizes to help diagnose prompt cache behavior. The new version preserves this:
- Still calls `log_info` with the "System prompts built" message
- Still calls `log_dim` for each role's byte count
- Uses `echo -n | wc -c` (same as the old version) to measure prompt sizes

### Execution Context vs. Review Standards

The old function bundled "Review Standards" (lines 521–554) as a separate heredoc appended to the Review prompt. This content is now part of `scripts/prompts/review/overlay.md`, so it flows in via Layer 3. The semantics are preserved; the content is in a different file.

## Safety Harness

The `verify-semantic-equivalence.sh` script is the safety harness for this destructive refactor. It runs automatically when you run the verification in Step 4 above. If it fails, the script tells you exactly which lines are missing, so you can fix them before committing.

**If you accidentally delete a heredoc line that contains important content**, the verification script will catch it with a diagnostic like:
```
FAIL: semantic equivalence check

SM: PASS (365 significant lines checked)
DEV: FAIL (390 significant lines checked, 5 missing)
  - "Do not add a dependency to src/package.json..."
  - "...more missing lines..."
REVIEW: PASS (276 significant lines checked)
```

This tells you that the Dev prompt is missing 5 lines. You can then check the dev overlay prompt file and fix it.
