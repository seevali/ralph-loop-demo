## Overview

Build a re-runnable verification script that ensures the refactored prompt system (stories 1.1–1.3) captures all content from the original hardcoded heredocs. This is **not** a code change to the loop itself; it is a test artifact that will be re-run in story 1.5 after the heredocs are deleted to confirm nothing was lost.

**Key insight:** The original scope specified a byte-diff check, but that is architecturally impossible. The new architecture intentionally reorders the layers (Layer 1: Execution Context BEFORE Layer 2: BMAD persona), which is a feature — Layer 1's "do not HALT" override must win over contradictory instructions in the persona. Therefore, the two assemblies cannot be byte-identical. The actual goal — *verify that 1.5 doesn't accidentally drop content* — is achievable through **semantic equivalence**: every significant content line from the original heredocs must be present somewhere in the refactored output, regardless of order or formatting.

### 4. Semantic Equivalence Check (Core Logic)

For each role (SM, Dev, Review):

- ✅ Extract all **significant** lines from the baseline (pre-refactor).
- ✅ Significant = non-empty AND not a section header (`## ...`) AND not a separator (`---`) AND not a stub heading (`# Agent Persona`).
- ✅ For each significant line, verify it appears **as a substring** somewhere in the corresponding layered output.
- ✅ Collect any missing lines and report them.

**Definition of "significant line":**
- Non-empty line (after trimming leading/trailing whitespace)
- Does NOT match regex `^##\s+` (section header)
- Does NOT match regex `^---\s*$` (separator)
- Does NOT match regex `^#\s+Agent Persona` (stub heading)

**Substring matching logic:**
```bash
# For each significant line in baseline, check if it appears in layered output
while IFS= read -r line; do
  if [[ -z "$line" ]] || [[ "$line" =~ ^##[[:space:]] ]] || [[ "$line" =~ ^---[[:space:]]*$ ]]; then
    continue  # Skip insignificant lines
  fi
  
  if ! grep -qF "$line" "$layered_output_file"; then
    missing_lines+=("$line")
  fi
done < "$baseline_file"
```

### How to Extract Pre-Refactor Output

The script must source `scripts/ralph-loop.sh` and call `build_system_prompts()`. This function populates three variables: `SYSTEM_PROMPT_SM`, `SYSTEM_PROMPT_DEV`, `SYSTEM_PROMPT_REVIEW`.

After sourcing, call it cleanly:


### How to Parse `--dry-run-prompts` Output

Story 1.3 specifies that `--dry-run-prompts` prints the prompts with clear delimiters (e.g., `=== SM ===`, `=== DEV ===`, `=== REVIEW ===`). The script should use these delimiters to extract the three roles.

Example parsing:
```bash
local dry_run_output
dry_run_output=$(./scripts/ralph-loop.sh --dry-run-prompts 2>&1)

# Extract SM prompt (between "=== SM ===" and the next "===")
local sm_prompt
sm_prompt=$(echo "$dry_run_output" | sed -n '/^=== SM ===/,/^=== [A-Z]*/p' | sed '1d;$d')

# Similar for DEV and REVIEW
```

### Significant Line Filtering

A line is **significant** if:
- Non-empty (or non-whitespace-only after trim)
- Does NOT start with `##` (section header)
- Does NOT match `^---\s*$` (separator line)
- Does NOT match `^# Agent Persona` (specific stub heading)

Use Bash regex or simple string checks:
```bash
is_significant() {
  local line="$1"
  [[ -n "$line" ]] && \
  [[ ! "$line" =~ ^##[[:space:]] ]] && \
  [[ ! "$line" =~ ^---[[:space:]]*$ ]] && \
  [[ ! "$line" =~ ^#[[:space:]]+Agent[[:space:]]+Persona ]]
}
```

### Substring Matching

Use `grep -qF` (fixed-string, quiet mode) to check if a significant line from the baseline exists in the layered output:
```bash
if ! grep -qF "$significant_line" "$layered_output_file"; then
  missing_lines+=("$significant_line")
fi
```

The `-F` flag treats the search string as a literal fixed string (not a regex), which is safer for arbitrary text.

## Edge Cases & Gotchas

### 1. Whitespace Differences

Baseline and layered outputs may have different line breaks, indentation, or spacing. The substring check handles this by matching exact text, but be aware:
- If a baseline line has trailing spaces, the grep must match trailing spaces.
- If a layered line has reformatted indentation, the check will fail (substring won't match).

**Mitigation:** Normalize whitespace before comparison if needed (e.g., trim trailing spaces from all lines).

### 2. Shell Expansion in Heredocs

Bash heredocs expand variables (e.g., `$SHELL`, `$(...)`) unless quoted (`<<'EOF'`). Inspect the heredocs in `scripts/ralph-loop.sh` to see if any variables are expanded. If so, the pre-refactor capture will reflect the expanded values, and the layered output must match.

### 3. Multi-Line Content in Significant Lines

A "significant line" is a single newline-delimited line in the file. If the baseline has multi-line content (e.g., a code block), each line is checked individually. The substring matching should still work, but be aware that a multi-line block will be split into separate checks.

### 4. Non-Deterministic Order in Layered Output

The `load_prompt_layers()` function assembles Layer 1 + Layer 2 + Layer 3 in that order. The order should be deterministic (same every time), but if story 1.2 or 1.3 introduces any randomness (e.g., reading a directory without sorting), the layered output could vary. Ensure the layer loader is deterministic.

### 5. Error Handling in Captures

If `capture_baseline()` fails (e.g., `build_system_prompts()` errors), the script should exit 1 with a clear error message. Same for `capture_layered()` if `--dry-run-prompts` fails.
