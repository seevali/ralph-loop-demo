## Escape handling (design-correct)

The script stores this text inside double-quoted bash assignments, escaping `\"`
and `` \` ``. The extracted markdown files store the **unescaped** characters
(real backticks, real double-quotes), matching the story spec's content blocks
and the implementation summary's stated decision — the loader (story 1.2) reads
files as-is with no unescaping step. Confirmed on `project-conventions.md` lines
5 and 14 vs script 387/396, and `review/overlay.md` line 18 vs script 473.

## Non-blocking observations (do NOT act on for this story)

1. **Trailing newlines.** Each extracted file ends with a single trailing
   newline (standard MD convention), whereas the script's `common` and review
   strings end without one (the closing `"` follows the last word). Per-section
   content is verbatim, so this is not an AC violation for 1.1. Flagging only so
   the story 1.2 developer designs the read step to strip trailing newlines
   before joining layers with `---` separators, and so the story 1.4 byte-diff
   compares loader output (not raw files) against the original.

2. **Review fallback header is intentionally omitted — verify the loader
   reproduces the source asymmetry.** `bmad-fallbacks/review.md` stores only the
   bare stub; the `# Agent Persona` header that wraps it in the script (lines
   447–449) is deferred to the loader (1.2). This is a reasonable choice and the
   stub text itself is correct, so it does not block 1.1. But note the source is
   *asymmetric*: the review fallback in `ralph-loop.sh` **includes** the
   `# Agent Persona` header (line 447), while the SM and Dev fallbacks (lines
   416, 431) do **not**. So "consistent with the SM/Dev fallback pattern" only
   holds for the stored file contents — not for the reassembled output. For story
   1.4's byte-diff to pass, the loader must add `# Agent Persona` for the review
   fallback (and all persona cases) but **not** for the SM/Dev fallback cases.
   The README's loader pseudocode (lines 62–70) currently shows no
   `# Agent Persona` injection at all. The story 1.2 developer should treat
   header reconstruction as explicit, role-aware work; story 1.4 is the gate
   that confirms it. No action required in 1.1.
