## Key Implementation Decisions

### 1. Template placeholder naming (`{{UPPER_CASE}}`)
All three new templates use `{{APP_DIR}}`, `{{CHECKPOINT_COMMAND}}`, and `{{STACK_DESCRIPTION}}`. The `project-conventions.md` template only uses `{{CHECKPOINT_COMMAND}}` and `{{STACK_DESCRIPTION}}` (no APP_DIR), as specified.

### 2. Plan field mapping
The wizard (Story 2.2) produces `targetDir` and `classification` — not `targetPath`/`targetClass` as the story spec's reference shape uses. The writer uses the actual wizard output field names throughout and adds `force`/`yes` from CLI options in `bin/ralph.js`.

### 3. Ownership classification
Encoded in `getOwnership(filePath)` as a pure function:
- **`installer-owned`:** `scripts/ralph-loop.sh`, `scripts/prompts/**`, `docs/project-conventions.md`, `.gitignore`
- **`user-owned`:** everything else (scaffold docs under `docs/epics/`, future user content)

This means scaffolded PRD and stories files start as `user-owned` — the installer will never overwrite them after the initial write, even on re-run with `--force`.

### 4. Conflict confirmation modes
The `yes`/`force` flags drive three distinct code paths in `confirmConflicts()`:
- `yes=false` → interactive (per-file prompt via `@clack/prompts`; injectable for testing)
- `yes=true, force=false` → non-interactive: print conflicts and `exit(1)`, no writes
- `yes=true, force=true` → non-interactive force: overwrite installer-owned, skip user-owned with a warning

### 5. Checksum stability
`hashString()` normalizes CRLF → LF before hashing, and `hashFile()` calls `hashString()`. This ensures checksums are consistent across Windows and Unix even if an editor adds CRLF line endings.

### 6. Atomic writes
Both `executeWrite()` and `writeManifest()` use a temp-file + `fs.rename()` pattern (`.ralph-tmp` suffix). The manifest is written **last** (Phase 5) so that a partial install leaves no manifest — on re-run the installer defaults to fresh-install conflict detection.

### 7. Rollback on write failure
`executeWrite()` tracks written files and deletes them on error. It does not restore originals (no pre-backup), since the manifest is absent and a clean re-run will re-prompt. This avoids the complexity of a full rollback while keeping the filesystem clean.

### 8. `.gitignore` append logic
`buildGitignoreContent()` reads the existing `.gitignore` (if present), filters out already-present entries, and merges new ones under a `# Ralph Loop` section. If all entries already exist, it returns `null` and the file is omitted from the write map entirely (no spurious conflict).

### 9. E2E tests at integration level
The story spec notes AC-5 bash fixtures are part of Story 4.1. This story covers AC-5 at the integration level: `writeInstall()` is called directly against a real temp directory, verifying that the complete pipeline (build map → detect conflicts → execute → manifest) produces a consistent, re-runnable state.
