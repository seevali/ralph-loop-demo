## Summary

Implement the write engine (`installer/src/writer.js`) that takes an in-memory `InstallPlan` (output from Story 2.2's wizard), renders template files by substituting the user's project facts, detects file conflicts using a default-deny policy, confirms changes with the user before writing, and records installation metadata in `.ralph/manifest.json`. This is the stage that transforms the plan into files on disk.

### 2. File Rendering and Substitution

#### 2.1 Template Files to Render

The writer must process three template files from `installer/templates/loop/`:

1. **`epic-stub-prd.md`** → `docs/epics/project-prd.md`
   - Substitute `{{APP_DIR}}` with user's `appDir` (default "src")
   - Substitute `{{CHECKPOINT_COMMAND}}` with user's `checkpointCommand`
   - Substitute `{{STACK_DESCRIPTION}}` with user's `stackDescription`
   - Include teaching comments (already in template)

2. **`epic-stub-stories.md`** → `docs/epics/project-stories.md`
   - Substitute `{{APP_DIR}}`, `{{CHECKPOINT_COMMAND}}`, `{{STACK_DESCRIPTION}}`
   - Verify all story headers match the strict `### Story X.Y: Title` format (used for loop parsing)
   - Keep all teaching comments intact

3. **`project-conventions.md`** → `docs/project-conventions.md`
   - Substitute `{{CHECKPOINT_COMMAND}}` and `{{STACK_DESCRIPTION}}`
   - This is a core reference doc; must render cleanly

**Rendering implementation:**

```javascript
// pseudocode
function renderTemplate(templateContent, substitutions) {
  let rendered = templateContent;
  for (const [key, value] of Object.entries(substitutions)) {
    const placeholder = `{{${key}}}`;
    rendered = rendered.replaceAll(placeholder, value);
  }
  return rendered;
}

// Verify all placeholders are substituted
function validateNoUnsubstituted(content) {
  const unreplaced = content.match(/\{\{[A-Z_]+\}\}/g);
  if (unreplaced) {
    throw new Error(`Unsubstituted placeholders: ${unreplaced.join(', ')}`);
  }
}
```

#### 2.2 Conditional File Writes

Depending on `taskSource`:

- **If `taskSource === 'scaffold'`:**
  - Render and write `docs/prd.md` from scaffold template
  - Render and write `docs/epics/` stubs from scaffold template
  
- **If `taskSource === 'existing'`:**
  - Do NOT write `docs/prd.md` or `docs/epics/`
  - User points at existing files; we only validate they exist

### 3. Conflict Detection and Default-Deny Policy

#### 3.1 Conflict Classes

A conflict occurs when any file the writer intends to write already exists **and is not in the manifest** (or the manifest classifies it as `user-owned`).

Partition files into two classes in the manifest:

- **`installer-owned`**: Files the installer manages and may overwrite during updates. Examples:
  - `scripts/ralph-loop.sh`
  - `scripts/prompts/**`
  - `.gitignore` entries (appended only; never truncate)
  
- **`user-owned`**: Files created by the user or the app that the installer never overwrites. Examples:
  - `docs/prd.md` (if user edited after install)
  - `docs/epics/` (if user added stories)
  - `src/` (Demo Track app code)
  - `.ralph/manifest.json` itself

#### 3.2 Conflict Detection Algorithm

```javascript
function detectConflicts(plan, writeMap) {
  const conflicts = [];
  
  for (const [filePath, content] of writeMap) {
    const fullPath = path.join(plan.targetPath, filePath);
    const exists = fs.existsSync(fullPath);
    
    if (!exists) continue; // No conflict
    
    // Load manifest if it exists
    const manifest = loadManifest(plan.targetPath);
    
    if (!manifest) {
      // Fresh install: no manifest yet, so any existing file is a conflict
      conflicts.push({
        path: filePath,
        reason: 'file-exists-no-manifest',
        existingChecksum: hashFile(fullPath),
      });
      continue;
    }
    
    const ownership = manifest.files[filePath]?.ownership;
    
    if (ownership === 'user-owned') {
      // User owns this file; never overwrite
      conflicts.push({
        path: filePath,
        reason: 'user-owned',
      });
    } else if (ownership === 'installer-owned') {
      // Installer may overwrite; check if locally modified
      const existingChecksum = hashFile(fullPath);
      const manifestChecksum = manifest.files[filePath]?.checksum;
      
      if (existingChecksum !== manifestChecksum) {
        // File was modified since last install
        conflicts.push({
          path: filePath,
          reason: 'locally-modified',
          existingChecksum,
          manifestChecksum,
        });
      }
      // else: file is unchanged from last install, OK to overwrite
    }
  }
  
  return conflicts;
}
```

#### 3.3 User Confirmation

If conflicts exist:

1. **Display all conflicts** with clear reason (which class, why it's a conflict)
2. **Default-deny:** Do NOT proceed without explicit action
3. **Interactive mode** (`--yes` not set):
   - Prompt per-file: "File X is [reason]. Keep mine / Take new / Back up and take new?"
   - Only proceed if user approves all
4. **Non-interactive mode with `--force`**:
   - Replace conflicting `installer-owned` files
   - Skip `user-owned` files (warn but don't error)
5. **Non-interactive mode without `--force`**:
   - Print all conflicts and exit non-zero
   - Do NOT prompt

### 4. Manifest Structure (`.ralph/manifest.json`)

Write `.ralph/manifest.json` with this schema:

```json
{
  "version": "string (semver, from package.json)",
  "installedAt": "ISO-8601 timestamp",
  "updatedAt": "ISO-8601 timestamp (on update)",
  
  "files": {
    "scripts/ralph-loop.sh": {
      "ownership": "installer-owned",
      "checksum": "sha256:hex-string",
      "path": "scripts/ralph-loop.sh"
    },
    "docs/prd.md": {
      "ownership": "user-owned",
      "checksum": "sha256:hex-string (for conflict detection)",
      "path": "docs/prd.md"
    }
    // ... all written files
  },
  
  "wizardAnswers": {
    // All Q&A from the wizard, for reference on re-run
    "projectName": "user's answer",
    "appDir": "src",
    "checkpointCommand": "npm run build && npm test",
    "stackDescription": "React 19 + Vite + TypeScript",
    // ... all other answers
  },
  
  "targetClass": "empty | existing-project | existing-install",
  "installedVersion": "version from prior install (if update)"
}
```

**Checksum calculation:**

```javascript
function fileChecksum(filePath) {
  const content = fs.readFileSync(filePath);
  return 'sha256:' + crypto.createHash('sha256').update(content).digest('hex');
}
```

### 5. Implementation Steps (Phased)

#### Phase 1: Rendering Engine (`installer/src/writer.js` stub)

1. **Create `installer/src/writer.js`** with export:
   ```javascript
   export async function writeInstall(plan) {
     // Phase 1: build write map (what files to write and their content)
     // Phase 2: detect conflicts
     // Phase 3: confirm with user
     // Phase 4: write to disk
     // Phase 5: write manifest
   }
   ```

2. **Implement phase 1: Build the write map**
   - Load all template files from `installer/templates/loop/`
   - Render each template with substitutions (appDir, checkpointCommand, stackDescription)
   - Validate no unsubstituted placeholders remain
   - Populate a Map: `{ filePath → content }`
   - Include system files: `.gitignore`, `CLAUDE.md` (if absent), etc.

#### Phase 2: Conflict Detection

3. **Implement `detectConflicts()` function** (see 3.2 above)
   - Load existing manifest (if present)
   - Walk the write map and check each file against filesystem
   - Return list of conflicts with reason and metadata

#### Phase 3: User Confirmation

4. **Implement `confirmConflicts()` function**
   - If no conflicts, return `approved = true`
   - If `--force` and non-interactive, approve installer-owned, skip user-owned
   - If interactive, prompt per-file using `@clack/prompts` (same library as wizard)
   - Return approved set of files to write

#### Phase 4: Disk Writing

5. **Implement `executeWrite()` function**
   - Create target directory structure if absent
   - Write each approved file (atomically if possible)
   - On write error, rollback all partial writes and throw

#### Phase 5: Manifest

6. **Implement `writeManifest()` function**
   - Compute checksums for all written files
   - Classify each file as `installer-owned` or `user-owned`
   - Write `.ralph/manifest.json` with all metadata
   - Ensure `.ralph/` directory exists; never commit it

## Edge Cases and Gotchas

### 1. Unresolved Template Placeholders
**Gotcha:** If the wizard passes an empty `stackDescription`, rendering might leave `{{STACK_DESCRIPTION}}` blank but not fail.  
**Mitigation:** `validateNoUnsubstituted()` must catch ANY unreplaced `{{...}}`, even if the substitution value is empty string. Use a default ("Unknown stack") if wizard doesn't provide a value.

### 2. File System Race Conditions
**Gotcha:** Between detecting conflicts and writing, another process could create or delete files.  
**Mitigation:** Accept this as unavoidable in a non-atomic install. Log a warning if a write fails and a file disappeared; don't treat it as fatal.

### 3. Path Normalization
**Gotcha:** User provides relative path like `../myproject`; absolute resolution varies by cwd at run time.  
**Mitigation:** Resolve all paths to absolute at plan validation time (in wizard or early in writer). Use `path.resolve()` before any fs operations.

### 4. Checksum Stability Across Platforms
**Gotcha:** CRLF vs LF line endings on Windows vs Unix could cause checksums to differ even though the user didn't edit the file.  
**Mitigation:** Normalize line endings (LF only) before hashing. Read files in text mode and write in consistent LF.

### 5. Manifest Ownership Classification
**Gotcha:** Is `docs/project-conventions.md` installer-owned (we write it) or user-owned (they might edit it later)?  
**Mitigation:** Per PRD FR-8, classify by mutation pattern: `installer-owned` if we write it and expect to overwrite on update; `user-owned` if user writes it or we don't expect to touch it again. For scaffolded PRDs/epics, initial write is `user-owned` (user will edit them). For `project-conventions.md`, it's `installer-owned` because we update it when loop knobs change.

### 6. `.gitignore` Appending vs Overwriting
**Gotcha:** We write a set of Ralph-specific entries to `.gitignore`. If the user has existing entries, should we overwrite or append?  
**Mitigation:** Always append. Read existing `.gitignore` (if present), add our entries only if not already present, write the merged result. Mark in manifest as `installer-owned` (we may re-run and update entries).

### 7. Concurrent Installs / Updates
**Gotcha:** Two invocations of the installer running simultaneously on the same target.  
**Mitigation:** No explicit locking. Accept that concurrent installs are unsupported (document in README). Manifest is written last, so one install will clobber the other's manifest; files are written atomically, so the filesystem isn't corrupted.

### 8. Windows Path Separators
**Gotcha:** `path/to/file` in the write map vs `path\to\file` in manifest on Windows.  
**Mitigation:** Use `path.normalize()` and `path.sep` consistently. Store paths in manifest using forward slashes (canonical); convert on read for filesystem operations.

## Implementation Notes for the Developer

1. **Use Node.js built-ins only** for file I/O (`fs.promises`), hashing (`crypto`), and paths (`path`). No external libraries beyond those already in `package.json`.

2. **Separate rendering logic from I/O logic** so rendering can be unit-tested without touching the filesystem.

3. **Write manifest last.** If anything fails before manifest write, the install is considered incomplete. On re-run, there's no manifest to guide conflict resolution, so we default-deny again.

4. **Log all decisions:** which files are being written, which conflicts were found, what the user approved. Help the user understand what happened.

5. **Integrate with Story 2.2's wizard:** The writer should be called with the InstallPlan returned from the wizard. See Story 2.2 for the signature of `wizard()`.

6. **Reference previous story code:** Study `installer/src/classify.js` (Story 2.1) for patterns on loading and parsing the manifest, and `installer/src/preflight.js` (Story 1.4) for how to output structured checklist-style messages.

7. **Handle the update scenario explicitly:** When manifest exists and `targetClass === 'existing-install'`, the writer is in update mode. This story doesn't fully implement updates (that's Story 3.2), but lay the groundwork: load the manifest, detect which files are installer-owned, and preserve checksums for later conflict detection.
