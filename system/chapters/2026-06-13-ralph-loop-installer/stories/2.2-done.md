## Key Implementation Decisions

### 1. Dependency injection via `opts.prompts` and `opts.exit`

The wizard accepts injectable prompt functions (`opts.prompts`) and an exit function (`opts.exit`). This mirrors the existing pattern in `preflight.js` (which injects `commandExists`) and makes all interactive behavior testable without spawning a real TTY or killing the test runner on `process.exit(0)`.

### 2. `isCancel()` return-value check — not try/catch

`@clack/prompts` returns the cancel symbol (`Symbol("clack:cancel")`) when the user presses Ctrl-C; it does not throw. The story spec's try/catch example is misleading. Each prompt call is wrapped by `checkCancel()` which calls `isCancel(value)` to detect cancellation.

### 3. Internal `CANCELLED` sentinel for clean cancellation flow

When the user cancels (via `isCancel()` or by declining a confirm prompt), the wizard:
1. Calls `p.outro('Nothing was changed.')` — shows the exit message
2. Calls `exit(0)` — terminates in production; records the call in tests
3. `throw CANCELLED` — stops execution in test mode (where `exit` is a no-op)

The outer `try/catch` catches `CANCELLED` and returns `undefined`. This avoids the problem of injected `exit()` being a no-op that lets subsequent code crash on missing values.

### 4. Validators exported separately for independent testing

All `text()` validate callbacks are named functions on an exported `validators` object (`validators.appDir`, `validators.checkpointCommand`, etc.). This allows direct unit testing without mocking the full wizard flow.

### 5. Non-TTY guard skipped when `opts.prompts` is injected

`if (!opts.prompts && !process.stdin.isTTY)` — the TTY check only fires in the real (non-test) path. Test code always provides `opts.prompts`, so the guard never blocks tests.

### 6. Confirm prompt count depends on classification — indices matter for tests

The wizard calls `confirm()` six times for `empty` classification, five times otherwise:

| Index | Prompt | `empty` | `existing-project` | `existing-install` |
|-------|--------|---------|--------------------|--------------------|
| `[0]` | Explainer ("Continue?") | ✓ | ✓ | ✓ |
| `[1]` | Target directory ("Continue?") | ✓ | ✓ | ✓ |
| `[2]` | Git init ("Initialize a git repository?") | ✓ | — | — |
| `[2/3]` | addGitignoreEntries | `[3]` | `[2]` | `[2]` |
| `[3/4]` | addNpmScripts | `[4]` | `[3]` | `[3]` |
| `[4/5]` | Final "Create this install?" | `[5]` | `[4]` | `[4]` |

Tests that override specific confirms pass the array via `{ confirms: [...] }`.  
The `runGitInit` boolean is stored in the `InstallPlan` (defaults to `false` for non-empty classifications).

## Review fixes applied (2026-06-13)

**Blocking issue (AC4):** The git init offer for empty directories was implemented as a static message only — no `confirm()` prompt was shown and no decision was stored in the `InstallPlan`. Fixed by:

1. **`installer/src/wizard.js`** — Added a `confirm()` prompt (Step 3b) after the target confirmation, shown only when `classification === 'empty'`. The answer is stored as `runGitInit: boolean` on the `InstallPlan`. Actual `git init` execution is deferred to Story 2.3 (AC2's zero-side-effects constraint preserved).

2. **`installer/src/wizard.test.js`** — Updated four tests whose confirm-call indices shifted for the `empty` classification (added one confirm at index 2). Added four new tests asserting: default `runGitInit=true` for empty dirs, opt-out path stores `false`, and the prompt is not offered for `existing-project` / `existing-install`.

**Non-blocking issue (ralph.js try/catch):** Added a try/catch around the `runWizard` call in `bin/ralph.js` so TTY errors and unexpected exceptions show a clean message and exit 1 instead of a raw stack trace.

Test count increased from 61 → 65 (4 new tests).

## What was NOT done (deferred by design)

- **AC1 / AC2 interactive manual verification** — requires a real TTY. The unit tests with mocked prompts cover the logic; E2E testing is deferred to Story 4.1 as specified.
- **`modelOrder` customization** — locked to `['opus', 'sonnet', 'haiku']`; Story 3.1 will parameterize via CLI flags.
- **Update semantics for `existing-install`** — wizard flows identically to a fresh install; nuanced keep/update/backup semantics are deferred to Story 3.2.
