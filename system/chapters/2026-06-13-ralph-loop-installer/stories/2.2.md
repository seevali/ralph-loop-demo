## Summary

Build the interactive wizard that walks users through configuring a Ralph Loop install, collecting all configuration into an in-memory `InstallPlan` data structure. The wizard uses `@clack/prompts` for a friendly, terminal-based UI with sensible defaults. **Critical:** no filesystem writes occur during the wizard — all changes are staged in memory and only committed by Story 2.3 (Write engine).

## Acceptance Criteria

### AC1: Wizard produces a complete InstallPlan from Enter-only interaction

- Run `npm run dev` or run the wizard interactively with all default prompts (only pressing Enter)
- Verify the wizard completes without error
- Verify the returned InstallPlan object contains all required fields (see "Data Structure" below) with sensible documented defaults
- No files are created or modified during the wizard run
- Example defaults: `appDir: "src"`, `checkpointCommand: "npm run build && npm test"`, `stackDescription: "React 19 + Vite + TypeScript"`, `loopRetries: 3`, etc.

### AC2: Ctrl-C anywhere in the wizard leaves zero filesystem effects

- Run the wizard and press Ctrl-C at each of these points: intro prompt, first question, midway through, last question
- Verify the wizard exits with a "nothing was changed" message or similar (output can be polished by Story 2.5)
- Verify `git status` shows no changes; no temporary files left behind
- No `.ralph/` directory is created
- Exit code should be 0 (cancellation is not an error, it's a user choice)

### AC3: Wizard output type-checks and validates

- The returned `InstallPlan` object conforms to the schema defined in "Data Structure" below
- All required fields are present
- All string fields are non-empty after trimming
- Loop configuration fields are integers >= 0 where applicable
- If the target directory was classified as `existing-install` (Story 2.1), the answers reflect an update context (e.g., skip prompts that are locked to the existing install)
- Validation happens at collection time (e.g., checkpoint command must be non-empty, not just when returning)

### AC4: Wizard integrates cleanly with preflight and classification

- After the user confirms the target directory from Story 2.1, the wizard has access to the classification result
- If classification returns `empty`, offer to run `git init` (preflight-like behavior, but wizard-initiated)
- If classification returns `existing-install`, adapt the wizard prompts to reflect update semantics (e.g., "Update this install?" instead of "Create a new install?")
- Preflight checks run before the wizard begins; the wizard assumes all preflight results are available

### AC5: Edge cases are handled gracefully

- User enters special characters, very long strings, or whitespace-only input → trimmed and re-prompted on validation failure
- User tries to set the checkpoint command to an empty string → re-prompted with an error message
- User's TTY session is broken mid-prompt (rare) → capture the error, show a fallback message, exit cleanly
- Stack description can reference multi-line conventions (e.g., "React 19 + Vite\nTypeScript strict mode\nCSS Modules") → properly escaped in the InstallPlan for rendering into markdown files

## Technical Details

### InstallPlan Data Structure

The wizard returns a plain JavaScript object with the following shape. Story 2.3 (Write engine) consumes this structure.

```javascript
{
  // Basic context
  targetDir: string,           // Absolute path to the target directory
  classification: 'empty' | 'existing-project' | 'existing-install',
  isUpdate: boolean,           // true if classification is 'existing-install'
  
  // Project facts
  appDir: string,              // e.g., "src" (relative to project root)
  checkpointCommand: string,   // e.g., "npm run build && npm test"
  stackDescription: string,    // Multi-line text describing the tech stack
  
  // Loop configuration (documented defaults visible to user)
  loopRetries: number,         // Default: 3
  maxTokensPerTurn: number,    // Default: 200000 (or derived from PRD)
  modelOrder: string[],        // Default: ['opus', 'sonnet', 'haiku']
  
  // Task source
  taskSource: 'scaffold' | 'existing',
  taskSourcePath?: string,     // Path to existing PRD/epic (if taskSource === 'existing')
  
  // Extras
  addGitignoreEntries: boolean, // Default: true
  gitignoreEntries: string[],   // Default: ['_bmad/', '.claude/skills/', 'scripts/logs/']
  addNpmScripts: boolean,       // Default: true (add 'dev-story', 'code-review' scripts?)
  npmScriptNames: string[],     // Default: ['dev-story', 'code-review']
  
  // Wizard answers for Story 2.3's manifest recording
  wizardAnswers: {
    explainerConfirmed: boolean,
    stackConfirmed: boolean,
    loopKnobsConfirmed: boolean,
    // ... any other y/n answers the wizard asks
  },
  
  // For Story 2.5's summary and next-steps
  summaryLines: string[],      // Human-readable lines summarizing the plan
}
```

### Implementation Steps

#### Step 1: Create the wizard module skeleton

Create `installer/src/wizard.js`:

```javascript
import { intro, outro, text, confirm, select, multiselect, spinner } from '@clack/prompts';
import { cancel, isCancel } from '@clack/prompts';
import picocolors from 'picocolors';

export async function runWizard(targetDir, classification, preflightResults) {
  // targetDir: absolute path to the target directory
  // classification: result from Story 2.1's classify() function
  // preflightResults: result from Story 1.4's preflight checks
  
  try {
    // Steps below...
  } catch (error) {
    if (isCancel(error)) {
      outro(picocolors.yellow('Nothing was changed.'));
      process.exit(0);
    }
    throw error;
  }
}
```

Key points:
- Use `@clack/prompts` for all interactive prompts
- Wrap the entire flow in a try-catch that catches `isCancel(error)` and exits with code 0
- Always import `{ cancel, isCancel }` from `@clack/prompts` to handle Ctrl-C correctly
- Return the `InstallPlan` object at the end, or exit early on cancellation

#### Step 2: Intro explainer prompt

Display a brief, friendly intro:

```javascript
intro(picocolors.bold('Ralph Loop Guided Installer'));
const explainerText = `
Welcome! This wizard will set up the Ralph Loop — an agent-driven development loop
running Claude Code to build your app story by story.

We'll collect information about your project and the loop configuration.
You can press Enter to accept any default, or Ctrl-C to cancel (zero changes made).
`;
await confirm({
  message: explainerText + '\n\nContinue?',
  initialValue: true,
});
```

Store in `wizardAnswers.explainerConfirmed`.

#### Step 3: Target confirmation prompt

Based on the classification from Story 2.1, confirm the target directory:

```javascript
const targetMessage = classification === 'empty'
  ? 'This directory is empty. I can initialize git and create a skeleton.'
  : classification === 'existing-project'
  ? 'This is an existing project. I\'ll add Ralph Loop files without touching your code.'
  : 'This directory already has a Ralph Loop install. I\'ll update it.';

await confirm({
  message: `${targetMessage}\n\nTarget: ${targetDir}\n\nContinue?`,
  initialValue: true,
});
```

If the user cancels here, exit with "nothing was changed".

#### Step 4: Project facts prompts

Collect the three project facts. Each should have a sensible default and validation:

**4a. App directory:**

```javascript
const appDir = await text({
  message: 'Application source directory (relative to project root)',
  initialValue: 'src',
  validate: (value) => {
    const trimmed = value.trim();
    if (!trimmed) return 'Directory name is required.';
    if (trimmed.startsWith('/') || trimmed.includes('..')) {
      return 'Use a relative path (e.g., src, packages/ui).';
    }
    return true;
  },
});
```

**4b. Checkpoint command:**

```javascript
const checkpointCommand = await text({
  message: 'Checkpoint command (build + test; shown in loop prompts)',
  initialValue: 'npm run build && npm test',
  validate: (value) => {
    const trimmed = value.trim();
    if (!trimmed) return 'Checkpoint command is required.';
    return true;
  },
});
```

**4c. Stack description:**

This is a multi-line prompt describing the tech stack. It's shown in the loop's system prompts and rendered into `project-conventions.md`.

```javascript
const stackDescription = await text({
  message: `Stack description (tech choices, testing approach, etc.)
  [Press Ctrl+D to finish multi-line input, or press Enter twice]`,
  initialValue: 'React 19 + Vite + TypeScript (strict mode)\nCSS Modules for styling\nVitest + React Testing Library for tests\nFetch API for HTTP',
  validate: (value) => {
    if (!value.trim()) return 'Stack description is required.';
    return true;
  },
});
```

Note: `@clack/prompts` text inputs are single-line by default. For multi-line, you may need to accept Enter-to-submit and document that newlines can be entered via manual edits to the generated `project-conventions.md` afterward. Alternatively, use a placeholder and accept the default as-is. **Check the @clack/prompts documentation** for multi-line support — if unavailable, settle for single-line with a note about editing afterward.

Store all three in the `InstallPlan` object.

#### Step 5: Loop knobs prompts

These are loop configuration values that affect how the Ralph Loop runs. All have defaults and are explained inline.

```javascript
const loopRetries = await text({
  message: 'Max retries per agent invocation (recommended: 3–5)',
  initialValue: '3',
  validate: (value) => {
    const num = parseInt(value.trim());
    if (isNaN(num) || num < 0) return 'Enter a non-negative number.';
    return true;
  },
});

const maxTokensPerTurn = await text({
  message: 'Max output tokens per loop turn (recommended: 150k–300k)',
  initialValue: '200000',
  validate: (value) => {
    const num = parseInt(value.trim());
    if (isNaN(num) || num <= 0) return 'Enter a positive number.';
    return true;
  },
});

// Optional: model order could be a select/multiselect, but default is fine for now
const modelOrder = ['opus', 'sonnet', 'haiku']; // Fixed for now; Story 3.1 can parameterize
```

Store in the `InstallPlan`.

#### Step 6: Task source prompt

Ask whether to scaffold new PRD/epic stubs or point at existing ones:

```javascript
const taskSource = await select({
  message: 'How would you like to define the first task?',
  options: [
    {
      value: 'scaffold',
      label: 'Start with a template PRD and epic (recommended)',
      hint: 'Includes comments explaining the format',
    },
    {
      value: 'existing',
      label: 'Point to an existing PRD and epic',
      hint: 'Must follow the ### Story X.Y: Title format',
    },
  ],
  initialValue: 'scaffold',
});

let taskSourcePath;
if (taskSource === 'existing') {
  taskSourcePath = await text({
    message: 'Path to your PRD file (relative to project root)',
    initialValue: 'docs/prd.md',
    validate: (value) => {
      if (!value.trim()) return 'Path is required.';
      return true;
    },
  });
}
```

Store in the `InstallPlan`.

#### Step 7: Extras prompts

Ask about `.gitignore` entries and npm scripts:

```javascript
const addGitignoreEntries = await confirm({
  message: 'Add default .gitignore entries (_bmad/, .claude/skills/, scripts/logs/)?',
  initialValue: true,
});

const addNpmScripts = await confirm({
  message: 'Add npm scripts for the loop (dev-story, code-review)?',
  initialValue: true,
});

const gitignoreEntries = addGitignoreEntries
  ? ['_bmad/', '.claude/skills/', 'scripts/logs/']
  : [];

const npmScriptNames = addNpmScripts
  ? ['dev-story', 'code-review']
  : [];
```

Store in the `InstallPlan`.

#### Step 8: Summary and confirmation

Display a human-readable summary of the plan and ask for final confirmation:

```javascript
const summaryLines = [
  `Target directory: ${targetDir}`,
  `App directory: ${appDir}`,
  `Checkpoint command: ${checkpointCommand}`,
  `Stack: ${stackDescription.split('\n')[0]}...`,
  `Task source: ${taskSource}`,
  `Loop retries: ${loopRetries}`,
  `Max tokens: ${maxTokensPerTurn}`,
  addGitignoreEntries ? 'Will add .gitignore entries' : 'No .gitignore changes',
  addNpmScripts ? 'Will add npm scripts' : 'No npm script changes',
];

console.log('\n' + picocolors.gray('Plan summary:'));
summaryLines.forEach(line => console.log('  ' + line));

const finalConfirm = await confirm({
  message: 'Create this install?',
  initialValue: true,
});

if (!finalConfirm) {
  outro(picocolors.yellow('Nothing was changed.'));
  process.exit(0);
}
```

#### Step 9: Return the InstallPlan

Build and return the complete InstallPlan object:

```javascript
const installPlan = {
  targetDir,
  classification,
  isUpdate: classification === 'existing-install',
  appDir: appDir.trim(),
  checkpointCommand: checkpointCommand.trim(),
  stackDescription: stackDescription.trim(),
  loopRetries: parseInt(loopRetries, 10),
  maxTokensPerTurn: parseInt(maxTokensPerTurn, 10),
  modelOrder: ['opus', 'sonnet', 'haiku'],
  taskSource,
  taskSourcePath: taskSourcePath?.trim(),
  addGitignoreEntries,
  gitignoreEntries,
  addNpmScripts,
  npmScriptNames,
  wizardAnswers: {
    explainerConfirmed: true,
    stackConfirmed: true,
    loopKnobsConfirmed: true,
  },
  summaryLines,
};

return installPlan;
```

## Edge Cases & Gotchas

### Edge Case 1: User cancels at different points

**Scenario:** User presses Ctrl-C after the intro prompt.

**Expected:** Wizard exits immediately with "Nothing was changed." and code 0.

**Implementation:** The try-catch wraps the entire wizard, so any `isCancel()` exception is caught and handled uniformly.

### Edge Case 2: Stack description with special characters or newlines

**Scenario:** User enters multi-line stack description (if supported) or special markdown characters like `*`, `#`, etc.

**Implementation:** Store the string as-is in the `InstallPlan`. Story 2.3's rendering step will escape it appropriately when writing to markdown files.

**Gotcha:** If `@clack/prompts` doesn't support multi-line text inputs natively, document that in a comment and suggest users can edit `project-conventions.md` after install.

### Edge Case 3: Very long checkpoint command

**Scenario:** User enters a checkpoint command with pipes, redirects, and complex logic.

**Expected:** The full command is stored as-is and later written to `project-conventions.md` for the loop to display.

**Implementation:** No special handling needed — store the trimmed string.

### Edge Case 4: Existing install update context

**Scenario:** Classification is `existing-install` (Story 2.1 detected `.ralph/manifest.json`).

**Expected:** The wizard should either:
- Skip some prompts (e.g., "App directory is already set to X; keep it?"), or
- Offer to update them with new defaults

**Implementation:** For this story, treat the update case the same as a fresh install (the SM can split this into a separate story if needed). Story 3.2 will handle the nuanced "keep / update / back up" semantics for modified files.

### Edge Case 5: Non-TTY environment (piped input, CI)

**Scenario:** The wizard is invoked in a non-TTY environment (e.g., piped to a file, running in CI).

**Expected:** The wizard should gracefully degrade or fail fast with a helpful error.

**Implementation:** Story 3.1 (Non-interactive mode) will handle `--yes` and other flags for CI. For this story, assume TTY is available. Add a check at the top of `runWizard()`:

```javascript
if (!process.stdin.isTTY) {
  throw new Error(
    'Wizard requires an interactive terminal. Use --yes for non-interactive mode (Story 3.1).'
  );
}
```

### Edge Case 6: User selects "existing" task source but path is invalid

**Scenario:** User selects task source `existing` and provides a path that doesn't exist or isn't a valid PRD.

**Expected:** The wizard doesn't validate file existence (that's Story 2.3's job). Just accept and return the path. Story 2.3 will error if the file doesn't exist.

**Implementation:** No special validation in the wizard; let the writer step handle validation.

## Summary

This story builds the core interactive wizard for collecting all install configuration into an in-memory `InstallPlan`. The wizard:

- Uses `@clack/prompts` for a polished, beginner-friendly terminal UI
- Collects seven categories of input: project facts, loop knobs, task source, and extras
- Provides sensible defaults for every prompt (Enter-only path works)
- Handles cancellation cleanly (Ctrl-C → "nothing was changed", exit 0, no filesystem effects)
- Returns a complete, validated `InstallPlan` object ready for Story 2.3 to write
- Integrates with Story 2.1's classification and Story 1.4's preflight results

The wizard itself makes **zero filesystem changes** — all writing is deferred to Story 2.3's write engine, preserving the safety property that the entire install can be cancelled at any point before the final confirmation.
