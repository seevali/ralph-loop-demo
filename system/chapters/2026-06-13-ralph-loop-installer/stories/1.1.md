## Technical Details

### Stack Exception Justification

From the PRD (Constraints section):
> Node.js is permitted **only** inside `installer/` (rule amendment in Story 1.1); everything else stays Bash + Markdown.

The installer is a Node.js CLI package (using Commander, @clack/prompts, and picocolors) because:
1. The BMAD Method's official installer is also Node.js-based (`npx bmad-method install`)
2. Node.js is the natural choice for an `npx`-driven CLI
3. The Node.js exception is scoped **only** to the `installer/` directory; all loop logic, prompts, and system improvements remain Bash + Markdown

### Why the New Reviewer Checks Matter

**`cd installer && npm test`:**
- Validates that the installer package's test suite passes
- Prevents npm/Node.js-specific regressions (import errors, syntax, type errors)
- Will be wired into the checkpoint once the installer stories land
- Parallel to the existing `bash -n` syntax check for shell scripts

**`installer/scripts/sync-templates.sh --check`:**
- Validates that the sync script (Story 1.3) has successfully copied canonical loop files into the installer's template directory
- Detects drift: if someone modifies `scripts/ralph-loop.sh` without re-syncing, the check fails
- Ensures the installer distributes the current loop version, not a stale snapshot
- Non-zero exit on drift (can be detected programmatically by the review agent)

## Edge Cases & Gotchas

### Edge Case 1: "What about the `installer/package.json` directory structure?"

The `installer/` directory is created in Story 1.2 (Scaffold the installer npm package). This story (1.1) **only** amends the rules to permit it. By the time Story 1.2 lands, the rules in this story are already in place.

### Edge Case 2: "Does the installer get its own CLAUDE.md?"

No. The installer is a Node.js package, not a separate BMAD-orchestrated project. It has a `package.json` and test suite, but no `CLAUDE.md` or agent loop. The root `CLAUDE.md` documents the exception; no additional guidance files are needed.

### Edge Case 3: "What if someone adds Python or Go inside `installer/`?"

The hard block in the reviewer rules is `cd installer && npm test`, not a generic language check. If someone adds non-Node.js code, the `npm test` will likely fail (import errors, missing dependencies). The reviewer would block on that failure. Additionally, the review agent reading `system/CLAUDE.md` understands that Node.js is permitted **only** in `installer/`; it would flag violations of that constraint explicitly.

### Edge Case 4: "What if the sync-templates script doesn't exist yet?"

The reviewer's hard block is:
> `installer/scripts/sync-templates.sh --check` does not pass (when the sync-templates script exists).

The "(when the sync-templates script exists)" qualifier means reviewers should **not** fail on this check until Story 1.3 lands. In practice, during the first few story reviews (1.1, 1.2), the sync-templates script won't exist, so the check is moot. The Story 1.2 review will pass without checking sync-templates. The Story 1.3 review will verify the script works. From Story 1.4 onward, both checks are active gates.

**Recommendation:** The dev/review agents should interpret "when it exists" as a no-op if the script hasn't landed yet, and a hard gate once it does.
