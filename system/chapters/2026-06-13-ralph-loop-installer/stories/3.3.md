## Story statement

The installer must provide a safe, interactive uninstall subcommand that removes installer-owned files (per manifest) and offers to remove user-owned files left behind. The uninstall command respects the same `--yes` and `--force` flags as the install command, allowing fully automated uninstall for scripting and CI/CD. The `.ralph/` directory itself is removed if empty after the uninstall completes.

## Acceptance criteria

1. **Uninstall removes installer-owned files:** Running `npx <package> uninstall` (or `uninstall --force` to skip prompts) in a directory with a valid manifest removes all installer-owned files (scripts/ralph-loop.sh, scripts/prompts/**, docs/project-conventions.md, .gitignore entries). The manifest file itself (.ralph/manifest.json) is removed last.

2. **User-owned files are preserved and listed:** After removing installer-owned files, the uninstall command lists all remaining user-owned files (docs/epics/**, docs/prd.md, custom docs, etc.) and prompts the user to confirm their removal. Without `--yes` or `--force`, user-owned files are never touched.

3. **Non-interactive uninstall with --yes:** Running `npx <package> uninstall --yes` into a fixture with user-owned files exits 0, removes installer-owned files, skips the user-owned prompt, and preserves all user-owned files byte-identically.

4. **Non-interactive uninstall with --force:** Running `npx <package> uninstall --force` removes both installer-owned and user-owned files without prompting; exits 0 if successful.

5. **Empty .ralph/ directory removed:** After uninstall completes (whether with or without user-owned file removal), the `.ralph/` directory is removed if it is empty.

6. **No manifest, no uninstall:** Running `uninstall` in a directory with no .ralph/manifest.json exits 1 with a helpful error: "No Ralph Loop install found here. (Expected .ralph/manifest.json)"

7. **Plain output in non-TTY:** Uninstall output respects `NO_COLOR` and `process.stdout.isTTY`, producing plain text when piped or with NO_COLOR=1.

## Implementation overview

This story adds the `uninstall` subcommand logic. It reuses the conflict-detection and TTY-aware output utilities created in Stories 2.3 and 3.1. The uninstall flow is similar to an "inverse install": read manifest, separate files by ownership, remove installer-owned files, prompt about user-owned files, then clean up the manifest directory.

### Files to create or modify

1. **`installer/src/uninstall.js`** — New file. Core uninstall logic: load manifest, categorize files, remove installer-owned files, prompt for user-owned files, remove manifest and empty `.ralph/` directory.

2. **`installer/src/uninstall.test.js`** — New file. Unit tests covering all acceptance criteria with fixture directories.

3. **`installer/bin/ralph.js`** — Modify existing. Wire the `uninstall` subcommand to call the uninstall module; pass `--yes` and `--force` flags.

## Detailed implementation steps

### Step 1: Create `installer/src/uninstall.js` — Core uninstall module

This module implements the five-phase uninstall flow. It is called from the CLI with options `{ yes, force, targetDir }`.

**Exports:**
- `async function uninstall(options)` — Main entry point. Returns `{ success: boolean, message: string }`.
- `async function loadManifest(manifestPath)` — Loads and validates the manifest JSON.
- `function categorizeFiles(manifestEntries)` — Separates files into `installerOwned` and `userOwned` arrays.
- `async function removeFile(filePath)` — Removes a file with error handling; returns `{ success, error }`.
- `async function removeEmptyDir(dirPath)` — Removes a directory only if empty.

**Phase 1: Load and validate manifest**
- Read `.ralph/manifest.json` from the target directory.
- If not found, exit with code 1 and message: `"No Ralph Loop install found here. (Expected .ralph/manifest.json)"`
- If the manifest exists but is malformed, exit with code 1: `"Manifest is corrupted. Cannot safely uninstall."`
- Parse the manifest and extract the file list with ownership classes.

**Phase 2: Categorize files**
- Iterate through manifest entries and separate by ownership class (`installer-owned` vs `user-owned`).
- Installer-owned files (from Story 2.3): `scripts/ralph-loop.sh`, `scripts/prompts/**`, `docs/project-conventions.md`, `.gitignore`.
- All other files in the manifest are user-owned.

**Phase 3: Remove installer-owned files**
- For each installer-owned file, attempt to delete it.
- If deletion fails (permission denied, file already gone), log a warning but continue.
- Collect errors and report them after all attempts.
- If any installer-owned file cannot be removed and `--force` is not set, prompt to continue or abort.
- If `--force` is set, log warnings and continue regardless of errors.

**Phase 4: Prompt for user-owned file removal**
- If no user-owned files exist, skip to Phase 5.
- If `--yes` or `--force` is set:
  - With `--yes`: skip the prompt, preserve all user-owned files, print: `"[Skipped] 3 user-owned files preserved."`
  - With `--force`: skip the prompt, remove all user-owned files.
- If interactive (neither `--yes` nor `--force` set):
  - List all user-owned files with relative paths (e.g., `docs/prd.md`, `docs/epics/story-1.md`).
  - Prompt: `"Remove these files too? (y/n)"` (clack-styled if TTY, plain if not).
  - If user enters `y`, remove them.
  - If user enters `n` or presses Ctrl-C, preserve them and log: `"[Skipped] X user-owned files preserved."`

**Phase 5: Clean up manifest and .ralph/ directory**
- Remove the manifest file `.ralph/manifest.json`.
- Attempt to remove `.ralph/` directory.
- If removal fails (directory not empty), log a warning: `".ralph/ directory is not empty. Leaving it in place."`
- If removal succeeds, log: `"✓ Removed .ralph/ directory"`

**Output logging:**
- Use the `colors.js` utilities (Story 3.1) for TTY-aware output.
- Each successful removal: `"✓ Removed scripts/ralph-loop.sh"`.
- Each error: `"✗ Could not remove docs/file.md: permission denied"` (logged to stderr if TTY, stdout if not).
- At the end, print a summary: `"Uninstall complete. X files removed, Y files preserved."`.

**Error handling:**
- If no manifest is found, exit immediately with code 1.
- If a critical file (manifest itself) cannot be removed, exit with code 1.
- If non-critical files fail to remove, continue but collect errors; exit with code 1 if `--force` is not set.
- If user declines to remove user-owned files, exit with code 0 (success, but nothing removed).

**Implementation notes:**
- Use `fs.rmSync()` with `{ force: true, recursive: true }` to remove files and directories.
- For `.gitignore` entries, read the file, filter out the `# Ralph Loop` section and following entries (until the next section marker or EOF), and write back only if the file is not empty. If the file becomes empty, delete it.
- Cache the `isColorEnabled()` result from `colors.js` at the start of the function.

### Step 3: Modify `installer/bin/ralph.js` — Wire the uninstall subcommand

The `uninstall` subcommand already exists as a stub from Story 1.2. Update it to:

1. Accept the global flags: `--yes`, `--force`, `--directory`.
2. Validate that a target directory is provided (or default to cwd).
3. Call `uninstall({ targetDir, yes, force })`.
4. Exit with the appropriate code (0 on success, 1 on failure).
5. Print the return message to stdout (or stderr if error).

**Command signature:**
```bash
node ralph.js uninstall [options]

Options:
  -y, --yes         Preserve user-owned files without prompting
  -f, --force       Remove all files without prompting
  -d, --directory   Target directory (default: cwd)
  --help            Show help and exit
```

**Flow:**
```javascript
program
  .command('uninstall')
  .option('-y, --yes', 'Preserve user-owned files without prompting')
  .option('-f, --force', 'Remove all files without prompting')
  .option('-d, --directory <dir>', 'Target directory', process.cwd())
  .action(async (options) => {
    const result = await uninstall({
      targetDir: options.directory,
      yes: options.yes || false,
      force: options.force || false,
    });
    
    if (!result.success) {
      console.error(result.message);
      process.exit(1);
    } else {
      console.log(result.message);
      process.exit(0);
    }
  });
```

## Technical notes

### File removal order

Remove files in this order to avoid orphaned directories:
1. Individual installer-owned files (e.g., `scripts/ralph-loop.sh`, `scripts/prompts/**`).
2. User-owned files (if confirmed).
3. `.gitignore` file (if it's installer-owned).
4. `.ralph/manifest.json` (last, so a partial failure leaves evidence of what was installed).
5. `.ralph/` directory (if empty).

### .gitignore cleanup

When removing the `.gitignore` file, preserve entries that existed before the install. The manifest records the original `.gitignore` content (if it existed). To avoid data loss:
1. Read the manifest's original `.gitignore` content (if stored).
2. Read the current `.gitignore`.
3. Remove only the `# Ralph Loop` section and entries added by the installer.
4. Write back the remainder (which includes both pre-install and user-added entries).
5. If the file is empty after removal, delete it.

**Implementation detail:** Story 2.3's manifest stores the full `.gitignore` content before the installer modifies it. Use this to reconstruct the original state. If the manifest doesn't have this, conservatively read the current file and remove only the clearly marked Ralph Loop section (between `# Ralph Loop` and the next blank line or comment starting with `#`).

### Atomic cleanup

The `.ralph/` directory removal must happen last. If the uninstall is interrupted mid-way, the manifest will still be present, and a fresh uninstall can resume from the same point.

### Error recovery

If uninstall partially fails and the user re-runs the command:
- The manifest will guide the second run; files already deleted will be skipped.
- If the manifest itself is missing (e.g., manually deleted), treat it as "no install found" and exit with code 1.

### Permission handling

On systems where the installer doesn't have write permission to remove files (e.g., files installed as root):
- Without `--force`, prompt the user: `"Cannot remove <file>. Use --force to skip or --help for options."`
- With `--force`, log a warning and continue.

## Edge cases and gotchas

1. **Manifest structure changes:** If the manifest format changes in a future story (e.g., adding new fields or restructuring), ensure backward compatibility. Read the version field and apply different parsing logic if needed.

2. **.gitignore section markers:** The Ralph Loop section is marked by `# Ralph Loop` comment. Ensure the cleanup logic correctly identifies the start and end of the section. If a user manually edits the .gitignore and removes the marker, conservatively preserve the file (don't delete entries that might be important).

3. **Empty .ralph/ directory check:** Use `fs.readdirSync('.ralph')` to check if the directory is empty before removing it. Some systems may have hidden files (e.g., `.DS_Store` on macOS) that prevent deletion. Handle this gracefully.

4. **Cross-platform path handling:** Paths in the manifest are normalized (forward slashes). When comparing with `process.cwd()` or constructing file paths, ensure cross-platform compatibility (use `path.resolve()`, not string concatenation).

5. **Permission denied on manifest:** If the manifest file itself cannot be removed (rare, but possible on Windows with file locks), log a clear error and exit non-zero rather than leaving a zombie manifest.

6. **User aborts uninstall mid-prompt:** If the user presses Ctrl-C during the user-owned file prompt, the process exits without removing anything additional. Ensure the manifest is still present so a re-run can resume. Do not catch Ctrl-C — let it propagate.

7. **Non-interactive after --yes:** If a user runs `uninstall --yes` and it fails for some reason (e.g., permission denied on a critical file), the command should exit with code 1 and not attempt any cleanup. The `--yes` flag means "accept defaults" (preserve user files), not "ignore errors."

8. **Relative paths in prompts:** When listing user-owned files to the user, display relative paths from the target directory, not absolute paths. This is more readable and user-friendly.

9. **Race conditions:** If another process is modifying files in the target directory during uninstall, file removals may fail. Accept this as a limitation and log errors clearly.
