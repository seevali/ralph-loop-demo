## User Story

As a preflight checker in the wizard flow, I need to detect what state the target directory is in (empty, existing project, or existing Ralph install) so the wizard can offer contextually appropriate prompts and prevent overwriting user work.

---

## Acceptance Criteria

1. **Empty directory classification:** A directory with no files (or only `.`, `..`, `.git/`) classifies as `empty`.
2. **Existing project classification:** A directory with one or more project signals (`.git`, `package.json`, `Gemfile`, `pyproject.toml`, `*.sln`, `pom.xml`, `go.mod`, `Cargo.toml`, `composer.json`, `pubspec.yaml`) classifies as `existing-project` even if `.ralph/manifest.json` is absent.
3. **Existing Ralph install classification:** A directory with `.ralph/manifest.json` classifies as `existing-install` regardless of other signals.
4. **Ambiguous case:** A directory with both `.ralph/manifest.json` and missing required installer-owned files returns `existing-install` with an advisory note (used in Story 2.3 conflict detection, not blocking here).
5. **Unit tests:** At least four test cases covering empty, existing-project, existing-install, and ambiguous scenarios, each with a fixture directory created by the test setup.

---

## Implementation Plan

### Step 1: Create the classifier module (`installer/src/classify.js`)

**File:** `installer/src/classify.js`

This module exports a single async function `classifyTarget(dirPath)` that:

1. **Validates the path:** resolves to an absolute path, checks it exists
2. **Lists directory contents** (non-recursive, detecting hidden files)
3. **Applies classification rules in order:**
   - If `.ralph/manifest.json` exists → return `{ type: 'existing-install', signals: [...] }`
   - If any project signal file/folder detected → return `{ type: 'existing-project', signals: [...] }`
   - If empty → return `{ type: 'empty', signals: [] }`
4. **Returns a classification object** with:
   ```javascript
   {
     type: 'empty' | 'existing-project' | 'existing-install',
     signals: [ /* list of detected signal names */ ],
     hasManifest: boolean, // true if .ralph/manifest.json exists
     projectRoot: string   // absolute path to the directory
   }
   ```

**Signal detection logic:**

- **File signals:** `.git`, `package.json`, `Gemfile`, `pyproject.toml`, `pom.xml`, `go.mod`, `Cargo.toml`, `composer.json`, `pubspec.yaml`
- **Directory signals:** `.git/` (if present and is a directory)
- **Directory wildcard signals:** `*.sln` (any `.sln` file — MS Visual Studio solution)

Use Node's `fs.promises.readdir()` with `withFileTypes: true` for efficient type detection.

**Edge cases to handle:**

- Non-existent paths → throw with clear error message
- Permission denied on target directory → throw with guidance on permissions
- Symlinks to directories → resolve and classify the target
- Multiple signal matches → include all in the `signals` array, but classification logic doesn't change
- Race condition (file created between read and classification) → not a blocker for this story; consistency is best-effort

**Example behavior:**

```javascript
// Empty directory
await classifyTarget('/tmp/empty')
// → { type: 'empty', signals: [], hasManifest: false, projectRoot: '/tmp/empty' }

// Existing Node project
await classifyTarget('/tmp/node-app')
// → { type: 'existing-project', signals: ['package.json', '.git'], hasManifest: false, projectRoot: '/tmp/node-app' }

// Existing Ralph install
await classifyTarget('/tmp/ralph-install')
// → { type: 'existing-install', signals: ['.ralph'], hasManifest: true, projectRoot: '/tmp/ralph-install' }
```

## Technical Details

### Why these signals?

**Project signals** are drawn from the [BMAD preflight checks](../../../_bmad/core/src/modules/preflight/index.js) (if available) and common project structures:

- `.git` — any version-controlled project (Git)
- `package.json` — Node.js / npm projects
- `Gemfile` — Ruby projects
- `pyproject.toml` — Python projects (PEP 517/518)
- `pom.xml` — Java Maven projects
- `go.mod` — Go projects (Go modules)
- `Cargo.toml` — Rust projects
- `composer.json` — PHP projects
- `pubspec.yaml` — Dart / Flutter projects
- `*.sln` — Microsoft Visual Studio solutions (C#, C++, VB.NET)

This list prioritizes breadth (catches most common stacks) over exhaustiveness. Adding new signals in a future story (e.g., `Makefile`, `setup.py`, `gradle.build`) is low-cost.

### Why `.ralph/manifest.json`?

The manifest (created in Story 2.3) is the definitive marker of a Ralph Loop install. Checking for it first ensures we handle updates and re-runs correctly.

### Classification semantics

- **`empty`:** safe to proceed with first-time install; offer to run `git init`
- **`existing-project`:** proceed with *additive* changes; confirm each step in Story 2.2 wizard
- **`existing-install`:** enter update mode (Story 3.2); replace installer-owned files, preserve user files

## Edge Cases & Gotchas

### Case 1: Hidden files in "empty" directory

`.DS_Store` (macOS), `Thumbs.db` (Windows), and other OS-generated files should not prevent classification as `empty`. Solution: ignore known non-project files in the `signals` array; focus on `type` classification. For now, accept `.DS_Store` etc. as signaling "not empty" — a future story can refine this if needed.

### Case 2: `.git` as a directory vs. symlink (bare repos)

In a bare Git repository, `.git` is a directory; in a git worktree, `.git` might be a file. `fs.promises.readdir({ withFileTypes: true })` and a type check handle both.

### Case 3: Symlink cycles

`path.resolve()` doesn't follow symlinks; if a symlink points back to its parent, no infinite loop occurs. The classification still proceeds on the resolved path.

### Case 4: Very large directories

`fs.promises.readdir()` loads all entries into memory. For directories with tens of thousands of files, this could be slow. Acceptable for this story; if performance becomes an issue (large monorepos), a future story can add early-exit once a project signal is found.

### Case 5: Conflicting signals

Example: a Ralph install (`existing-install`) inside a parent Git repo (`existing-project`). The classification prefers `.ralph/manifest.json` (checked first), so it correctly returns `existing-install`. The `signals` array includes both `.git` and `.ralph` for transparency.

### Case 6: Permission denied on target

If the directory exists but is unreadable, `fs.promises.readdir()` throws. Catch and re-throw with a helpful message like: "Cannot read target directory. Check permissions: chmod +rx target-dir".
