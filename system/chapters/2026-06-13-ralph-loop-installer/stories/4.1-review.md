## Non-blocking observation (do not fix as part of this story)

`assertNoANSIEscapes` in `installer/test/assertions.js:201` uses
`/\x1b[\x5b\x5d]|[\x5b\x5d]/`. The second alternation `[\x5b\x5d]` matches a
bare `[` or `]`, so the helper actually rejects any output containing a literal
bracket, not just ANSI escapes. This is not blocking because:
(a) it cannot mask a real ANSI escape — every ANSI sequence contains `\x1b[`,
caught by the first alternation; (b) its only failure mode is a false positive,
and none occurs today (the `--yes` install output contains no brackets), so the
criterion-5a test passes and genuinely validates "no ANSI in non-TTY output".
Worth tightening to `/\x1b[\x5b\x5d]/` in a future hardening pass so the helper
matches its documented contract, but it does not affect any AC or gate now.
