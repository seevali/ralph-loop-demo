## Key Implementation Decisions

### `plan.skipBmad === false` strict check (not `!plan.skipBmad`)

`writeInstall()` guards the BMAD step with `plan.skipBmad === false` rather than `!plan.skipBmad`. This preserves backward compatibility: all existing writer tests pass plans where `skipBmad` is `undefined`, and with a strict-false check those plans correctly bypass the BMAD step — no network call, no test breakage. Plans produced by the wizard always set the field to an explicit boolean (`skipBmad: !installBmadStep`), so new installs work correctly.

### Dependency injection for `execFile` (not module mocking)

The codebase already uses DI throughout (`opts.log`, `opts.exit`, `opts.prompts`). Following that pattern, `installBmad` accepts `options.execFile` (a promisified child_process.execFile drop-in). This avoids ESM module mock complexity and makes tests deterministic without spawning real processes.

### Injectable `opts.installBmad` in `writeInstall()`

The writer integration tests inject a stub BMAD function, verifying the conditional call without triggering real `npx` runs. The default remains the real `installBmad` for production use.

### Non-TTY: noop spinner + plain-text log

When `isTTY` is false and no external spinner is provided, a noop spinner (object with empty `start`/`stop`) is used alongside `log('Installing BMAD modules...')`. This keeps the non-TTY output free of ANSI codes while still providing a progress line, satisfying AC-4.

### Non-fatal failure design

On `execFile` throw, the module catches the error, stops the spinner, prints the manual command + guidance via `log()`, and returns `{ success: true }`. The installer continues regardless of BMAD outcome.
