## Key Implementation Decisions

### 1. "keep" preserves the prior installer checksum in the manifest

After a "keep" decision, `writeUpdateManifest` does NOT update the checksum for that file — it stays as the original installer checksum. This ensures that the *next* update run still detects the file as locally modified (on-disk checksum != manifest checksum) and re-prompts the user. If we had recorded the user's checksum, the file would appear unmodified on the next run and the installer would silently overwrite it.

### 2. Backup extension is `.backup` (not `.bak`)

The existing interactive conflict resolution in `confirmConflicts` (Story 2.3) uses `.bak`. The update flow specified in Story 3.2 uses `.backup`. Both coexist without conflict.

### 3. `executeUpdate` builds the write map internally from the install plan

The install plan is reconstructed from `manifest.wizardAnswers` in `bin/ralph.js` before calling `executeUpdate`. This mirrors how `writeInstall` works (it also calls `buildWriteMap` internally). The wizard is not re-run in update mode — answers are preserved from the manifest.

### 4. User-owned files are excluded by `getOwnership()`, not the delta

`executeUpdate` calls the existing private `getOwnership()` function for each file in the write map. This is an absolute guard — user-owned files are skipped regardless of what the delta or conflict decisions say. The delta's `userOwned` array is not used to make write decisions (it's informational only).

### 5. `--update-conflicts` validated before writes in both the resolver and the CLI

`resolveConflicts` returns `{ succeeded: false, errors: [...] }` for invalid values — no decisions are made and no writes occur. `bin/ralph.js` also validates the flag before calling `detectUpdate` or any write function.

### 6. `.backup` overwrites silently with a warning log

If `<path>.backup` already exists when the user chooses "backup", `fs.rename` overwrites it (POSIX atomic rename semantics). A warning is logged: `"Warning: overwriting existing backup: <file>.backup"`. No error is thrown.
