## Objective

Write comprehensive end-to-end tests for the Ralph Loop installer, validating the complete install → doctor → update → uninstall workflows through non-interactive mode. Tests run as part of `cd installer && npm test` and directly verify each PRD success criterion (1–5).

## Dependencies

All functionality implemented by prior stories (1.1–3.3):
- Story 1.2: npm package scaffold with `npm test` runner (node:test)
- Story 1.3: sync-templates.sh script
- Story 1.4: preflight checks module
- Story 2.1: target classifier
- Story 2.2: wizard flow (InstallPlan builder)
- Story 2.3: writer engine, conflict handling, manifest
- Story 2.4: BMAD install step
- Story 2.5: doctor subcommand, GETTING-STARTED generation
- Story 3.1: non-interactive mode (`--yes`, conflict behavior, non-TTY output)
- Story 3.2: update flow (manifest detection, file preservation)
- Story 3.3: uninstall

## User Story

As a developer maintaining the Ralph Loop installer, I need comprehensive E2E tests so that I can confidently ship releases knowing the full install/update/uninstall cycle works correctly under real-world conditions (empty directory, existing projects, user modifications, automation pipelines).

## Acceptance Criteria

1. **E2E suite passes locally:** `cd installer && npm test` runs all E2E tests and exits 0.
2. **PRD criterion 1 (empty-dir E2E):** At least one test case validates non-interactive install into an empty directory, runs `doctor`, confirms all FR-6 files present, verifies epic stub parseability under story-header rules, and exits 0.
3. **PRD criterion 2 (Enter-only path):** At least one test case validates that the interactive wizard completed with only Enter presses (via `--yes`) produces a working install verified by `doctor`.
4. **PRD criterion 3 (update safety):** At least one test case demonstrates that modifying user-owned files (`docs/prd.md`, `scripts/prompts/common/project-conventions.md`) before re-running install preserves those files byte-identically after the update completes.
5. **PRD criterion 4 (sync gate):** At least one test case confirms `installer/scripts/sync-templates.sh --check` passes on a fresh install; a second test case mutates a synced file in the sandbox and verifies `--check` fails.
6. **PRD criterion 5 (non-TTY + conflict behavior):** At least one test case pipes `--yes` install output to capture and verifies zero ANSI escape codes. A second test case runs a conflicting install without `--force` and verifies exit code non-zero with no hanging.

## Technical Details

### Test Framework and Structure

- **Framework:** Node.js built-in `node:test` (established in Story 1.2)
- **Test files:** 
  - `installer/test/e2e.test.js` — orchestrates fixture setup, runs workflows, validates outputs
  - `installer/test/fixtures.js` — helper module providing temporary directory creation, cleanup, and standard fixture states
  - `installer/test/assertions.js` — helper module with domain-specific assertions (e.g., "epic stub is parseable", "manifest valid", "file unchanged")

### Test Fixtures

Two primary fixture types, both created as temporary directories cleaned up after each test:

#### Fixture 1: Empty Directory
- Initial state: empty, no `.git`, no `package.json`, no existing Ralph install
- Used by: empty-dir E2E (PRD criterion 1), Enter-only path (criterion 2), update safety (criterion 3)
- Teardown: remove temp dir and all contents

#### Fixture 2: Existing Project
- Initial state: contains `.git/` (stub), `package.json` (minimal), possibly other files to signal project type
- Used by: target classifier tests (already covered in Story 2.1, but E2E also validates)
- Teardown: remove temp dir and all contents

#### Fixture 3: Existing Ralph Install (for update tests)
- Initial state: empty dir → install with `--yes` → run doctor (verify pass) → record initial state
- Used by: update safety (criterion 3), uninstall (not a PRD criterion, but test for completeness)
- Teardown: remove temp dir and all contents

### Test Cases Mapped to PRD Criteria

#### Test 1: Empty-Dir E2E with Non-Interactive Install (PRD Criterion 1)

**Workflow:**
1. Create empty temp directory
2. Run `npx ./bin/ralph.js install --directory <temp-dir> --yes --app-dir src --checkpoint-cmd "npm run build"` (or minimal flags)
3. Run `npx ./bin/ralph.js doctor --directory <temp-dir>`
4. Verify exit code 0 for both commands
5. Verify all FR-6 files present in `<temp-dir>`:
   - `scripts/ralph-loop.sh`
   - `scripts/prompts/` (all subdirectories)
   - `docs/prd.md`
   - `docs/epics/` (with starter stubs)
   - `CLAUDE.md` (only if not pre-existing)
   - `GETTING-STARTED.md`
   - `.gitignore` (updated with ralph-specific entries)
   - `.ralph/manifest.json`
6. Parse `<temp-dir>/docs/epics/*.md` and verify each file contains valid `### Story X.Y: Title` headers (story-header rules from the loop parser)
7. Assert manifest contains correct checksums for all installer-owned files

**Expected output:** All assertions pass, exit code 0 from both install and doctor.

#### Test 2: Enter-Only Path with Interactive Wizard (PRD Criterion 2)

**Workflow:**
1. Create empty temp directory
2. Run installer with `--yes` flag (simulates Enter-only presses via automated acceptance of defaults)
3. Run `doctor` on the installed directory
4. Verify exit code 0 for both

**Note:** True interactive Enter-only testing is impractical in CI (requires spawning a tty, piping enters); `--yes` with defaults is the automation equivalent and sufficient per the PRD ("completed with only Enter presses" means accepting all defaults).

**Expected output:** All assertions pass, exit code 0 from both install and doctor.

#### Test 3: Update Safety with File Preservation (PRD Criterion 3)

**Workflow:**
1. Create empty temp directory
2. Install with `--yes` (creates initial state)
3. **Capture checksums** of user-owned files: `docs/prd.md`, `scripts/prompts/common/project-conventions.md` (via manifest)
4. **Mutate** these files with deterministic changes (append a comment line to each)
5. **Re-run install** with `--yes` in the same directory (update mode, detected by manifest presence)
6. Run update flow (manifest → version check → replace installer-owned only)
7. **Verify checksums:** read the same files again, confirm byte-identical to step 3 (user changes preserved)
8. Verify installer-owned files were replaced/updated (e.g., `scripts/ralph-loop.sh` should be current version)

**Expected output:** User-owned files unchanged, installer-owned files replaced, exit code 0.

#### Test 4: Sync Gate Pass (PRD Criterion 4, Part A)

**Workflow:**
1. Create empty temp directory
2. Install with `--yes`
3. Run `bash installer/scripts/sync-templates.sh --check`
4. Verify exit code 0

**Expected output:** Sync gate passes, exit code 0.

#### Test 5: Sync Gate Fail on Drift (PRD Criterion 4, Part B)

**Workflow:**
1. Create temp directory and install with `--yes`
2. **Mutate** a synced file in place: append a line to `<temp-dir>/scripts/ralph-loop.sh`
3. Run `bash installer/scripts/sync-templates.sh --check`
4. Verify exit code non-zero (failure)

**Expected output:** Sync gate detects drift, exits non-zero.

#### Test 6: Non-TTY Output (No ANSI Codes) (PRD Criterion 5, Part A)

**Workflow:**
1. Create empty temp directory
2. Run install with `--yes` but **redirect stdout to a file** (simulating piped output, non-TTY):
   ```bash
   npx ./bin/ralph.js install --directory <temp-dir> --yes > /tmp/output.txt 2>&1
   ```
3. Read captured output and search for ANSI escape codes (pattern: `\x1b\[` or `\033\[`)
4. Assert no ANSI codes found

**Expected output:** Output file contains no ANSI codes, install succeeds (exit 0).

#### Test 7: Conflict Detection Without `--force` (PRD Criterion 5, Part B)

**Workflow:**
1. Create temp directory with a conflicting file pre-existing (e.g., `scripts/ralph-loop.sh` or `CLAUDE.md` with different content)
2. Run install with `--yes` **without** `--force` flag
3. Verify exit code non-zero
4. Verify no writes occurred (directory state unchanged from step 1)
5. Verify the command did not hang (test timeout of 10s is sufficient)

**Expected output:** Install exits non-zero, no files written, test completes within timeout.

#### Test 8: Uninstall Workflow (Bonus, not a PRD criterion)

**Workflow:**
1. Create empty temp directory
2. Install with `--yes`
3. Run `npx ./bin/ralph.js uninstall --directory <temp-dir> --yes`
4. Verify installer-owned files removed (`scripts/`, `.ralph/manifest.json`)
5. Verify user-owned files listed but not removed (or confirm removal with `--yes` if that's the design)
6. Verify exit code 0

**Expected output:** Installer-owned files gone, directory state matches expectations.

## Implementation Gotchas and Mitigations

### Gotcha 1: Temporary Directory Cleanup on Windows
**Issue:** Windows may hold file locks, preventing directory removal if files are still open.
**Mitigation:** Ensure all file handles are closed before cleanup. Use `fs.promises` with explicit close or try-finally blocks. Add a short delay (100ms) before cleanup if needed.

### Gotcha 2: ANSI Code Detection Across Platforms
**Issue:** Different spinners/color libraries may emit different ANSI sequences.
**Mitigation:** Search for the most common patterns (`\x1b\[`, `\033\[`, `\\u001b[`). Test on both TTY and non-TTY to verify behavior.

### Gotcha 3: Sync Gate Test Must Use Real `sync-templates.sh`
**Issue:** The sync gate validates byte-for-byte match of synced files; a mock won't catch real drift.
**Mitigation:** Invoke `bash installer/scripts/sync-templates.sh --check` directly (not mocked); test operates on the actual installed files from Story 1.3.

### Gotcha 4: Update Mode Detection Fragile
**Issue:** Update mode depends on manifest presence and correct JSON structure; a malformed manifest may cause installer to default to install-from-scratch instead of update.
**Mitigation:** Verify manifest is valid after initial install (use `assertions.assertManifestValid`). Log manifest read/write in test for debugging.

### Gotcha 5: Non-TTY Spawning Must Suppress Stdout Interactivity
**Issue:** Even with `--yes`, the installer may attempt to detect TTY and change output mode if spawned in certain ways.
**Mitigation:** Pipe to a file explicitly and check the captured output. Verify with `tty` command during test if needed (e.g., `tty < /dev/null` should fail, redirected install should not attempt interactivity).

### Gotcha 6: Conflict Test Must Not Hang on User Prompt
**Issue:** If non-interactive mode fails to respect `--yes`/`--force` semantics, the installer may wait indefinitely for stdin.
**Mitigation:** Run conflict test with a short timeout (10s). If test hangs, it will fail the timeout. Log stderr to help diagnose which prompt is blocking.

### Gotcha 7: BMAD Install Step May Fail in Test Environment
**Issue:** `npx bmad-method install` in a test environment (no real npm credentials, possible network issues) may fail.
**Mitigation:** This is expected per Story 2.4 (BMAD step is skippable on failure). Tests should expect `doctor` to pass even if BMAD fails (only installer-owned files matter for E2E). If BMAD failure blocks the test, ensure Story 2.4 implements skippable behavior correctly.

### Gotcha 8: Epic Stub Parseability Regex May Be Too Strict
**Issue:** The loop parser's story-header regex is strict; if the stub generator creates invalid headers, tests will fail.
**Mitigation:** Verify in Story 2.3 that `project-conventions.md` template and stub generation produce valid headers. Test regex against known good/bad examples before relying on it in E2E.
