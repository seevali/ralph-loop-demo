## User Story

As a wizard orchestrator, I want to validate the installer's runtime environment before proceeding with interactive prompts, so that users encounter hard blockers early and receive actionable guidance without cryptic errors mid-flow.

## Context

The guided installer (FR-2 in [PRD](../prd.md#functional-requirements)) must validate the host environment and present results in a compact checklist before the wizard begins. This story implements the validation engine (`preflight.js`) and its test suite.

**Key design principle:** The checklist is a *notification* layer, not a decision-maker. Each check reports what it found; the wizard will decide which checks block installation vs. warn and continue. This story implements the checks only.

## Acceptance Criteria

- [x] **AC1 — Node.js hard fail:** Calling preflight reports Node.js version; if < 20, returns exit code 1 with a message pointing to node.js.org/download
- [x] **AC2 — git warn:** Calling preflight on a system without `git` reports "warn" status; includes suggestion to install via package manager
- [x] **AC3 — jq warn with hints:** Calling preflight reports jq status; if missing, includes per-OS install hint (`apt install jq`, `brew install jq`, `choco install jq`, `winget install jqlang.jq`)
- [x] **AC4 — claude CLI warn:** Calling preflight reports `claude` CLI status; missing or present—both are warnings, no block
- [x] **AC5 — bash detection:** On native Windows (no WSL2 bash), preflight exits with code 2 and prints WSL2 guidance; on other systems or with bash present, passes through
- [x] **AC6 — TTY checklist rendering:** On a TTY, checklist includes status glyphs (✓, ⚠, ✗) and color (via picocolors already available); on non-TTY, plain text lines only
- [x] **AC7 — Unit tests via PATH stubbing:** `cd installer && npm test` includes tests that:
  - Stub `PATH` to remove/add specific commands and verify each check's response
  - Test checklist rendering with `process.stdout.isTTY = true` and `= false`
  - Verify Node.js hard-fail exit code 1 when version < 20
  - Verify WSL2-detection exit code 2 on native Windows without bash
- [x] **AC8 — Return structure:** Preflight returns an object (or array) with:
  ```javascript
  {
    node: { status: 'pass' | 'fail', version: '20.x.x', message: '...' },
    git: { status: 'pass' | 'warn', message: '...' },
    jq: { status: 'pass' | 'warn', message: '...' },
    claude: { status: 'pass' | 'warn', message: '...' },
    bash: { status: 'pass' | 'warn' | 'fail', message: '...' },
  }
  ```
  where `fail` stops the installer (exit code 1 or 2), `warn` allows continuation, and `pass` is clear.

## Implementation

### Phase 1: Core Module (`installer/src/preflight.js`)

**1.1 Create the main module file**

File: `installer/src/preflight.js`

The module exports a single async function `preflight()` that:

1. **Detects Node.js version:**
   - Read from `process.version`
   - Parse major version (e.g., "v20.11.0" → 20)
   - Fail with exit code 1 if < 20
   - Return `{ status: 'fail', version: '...', message: '...' }` if fail, or `{ status: 'pass', version: '...', message: '...' }` if pass

2. **Checks for `git`:**
   - Use `which git` (or on Windows, `where git`) to locate the command
   - If found, `{ status: 'pass', message: 'git found' }`
   - If not found, `{ status: 'warn', message: 'git not found. Install: apt install git (Debian/Ubuntu), brew install git (macOS), or visit https://git-scm.com/download' }`

3. **Checks for `jq`:**
   - Use `which jq` (or `where jq` on Windows)
   - If found, `{ status: 'pass', message: 'jq found' }`
   - If not found, return a warn with per-OS hints:
     ```
     Debian/Ubuntu: apt install jq
     macOS: brew install jq
     Windows (Chocolatey): choco install jq
     Windows (winget): winget install jqlang.jq
     ```

4. **Checks for `claude` CLI:**
   - Use `which claude` (or `where claude` on Windows)
   - Always return warn status (never fail), even if not found:
     ```
     Pass: 'claude' CLI found
     Warn: 'claude' CLI not found. Install: npm install -g @anthropic-ai/claude-code or https://github.com/anthropics/claude-code
     ```

5. **Bash environment detection (Windows-specific):**
   - Detect if running on Windows (via `process.platform === 'win32'`)
   - If Windows: check if bash is available (`which bash` or `where bash` in Git Bash, WSL, MSYS, etc.)
   - If Windows + no bash: return `{ status: 'fail', message: '...' }` with WSL2 guidance and exit code 2
   - If non-Windows or Windows + bash found: return `{ status: 'pass', message: 'bash environment detected' }`

**Note on command detection:** Use the Node.js `child_process.execSync()` or `child_process.spawnSync()` to check for commands. Wrap in try-catch. Return the check object; let the caller decide what to do with the exit code.

**1.2 Add a render function for checklist output**

The module exports a second function `renderChecklist(results)` that takes the results object from `preflight()` and outputs:

- **On TTY (detected via `process.stdout.isTTY`):**
  ```
  ✓ Node.js v20.11.0
  ⚠ git not found — install via package manager
  ⚠ jq not found — macOS: brew install jq
  ✓ claude CLI not found (optional)
  ✓ bash environment detected
  ```
  (with picocolors for red, yellow, green where appropriate)

- **On non-TTY (piped or redirected):**
  ```
  NODE_VERSION: 20.11.0
  GIT: warn – git not found
  JQ: warn – jq not found
  CLAUDE: warn – claude CLI not found
  BASH: pass
  ```
  (plain text, no ANSI codes, no glyphs)

Respect `process.env.NO_COLOR` — if set, omit color even on TTY.

**1.3 Exit behavior:**

- Return the results object from `preflight()` for the caller to decide on exit code
- The function itself does NOT call `process.exit()`; let the caller (wizard or CLI) decide
- Document that caller must check `results.node.status === 'fail'` → exit 1, and `results.bash.status === 'fail'` → exit 2

**2.3 Command detection mocking strategy:**

For each test, override `child_process.execSync()` or `child_process.spawnSync()` to simulate command availability. Example:

```javascript
// In a test
const Module = require('module');
const originalRequire = Module.prototype.require;
Module.prototype.require = function(id) {
  if (id === 'child_process') {
    const cp = originalRequire(id);
    cp.execSync = jest.fn((cmd) => {
      if (cmd === 'which git') throw new Error('not found');
      return ''; // or return a version string
    });
    return cp;
  }
  return originalRequire(id);
};
```

Or use a simpler approach: export a testable version that accepts a `spawnFn` parameter so tests can inject mocks.

## Technical Details

### Command Detection

Use Node.js built-ins only; no external dependencies for command checking:

```javascript
const { execSync } = require('child_process');

function checkCommand(cmd) {
  try {
    execSync(`which ${cmd}`, { stdio: 'pipe' });
    return true; // found
  } catch {
    return false; // not found
  }
}
```

For Windows, use `where` instead of `which`:

```javascript
const checkCmd = process.platform === 'win32' ? 'where' : 'which';
execSync(`${checkCmd} ${cmd}`, { stdio: 'pipe' });
```

### Version Parsing

```javascript
const version = process.version; // e.g., "v20.11.0"
const major = parseInt(version.slice(1).split('.')[0], 10);
if (major < 20) {
  return { status: 'fail', message: '...' };
}
```

### Return Structure

Return a consistent object so the caller (wizard in Story 2.2) can inspect results:

```javascript
{
  node: { status: 'pass' | 'fail', version: '...', message: '...' },
  git: { status: 'pass' | 'warn', message: '...' },
  jq: { status: 'pass' | 'warn', message: '...' },
  claude: { status: 'pass' | 'warn', message: '...' },
  bash: { status: 'pass' | 'warn' | 'fail', message: '...' },
}
```

## Integration Notes

**How the wizard will use this:**
- Story 2.2 (wizard flow) will import and call `preflight()` immediately after target classification
- If `results.node.status === 'fail'`, the wizard exits with code 1 (hard stop, no recovery)
- If `results.bash.status === 'fail'`, the wizard exits with code 2 (WSL2 guidance, no recovery)
- Other warn statuses proceed with a confirmation prompt in the wizard (Story 2.2 scope)

**Caller responsibility:**
The caller (the CLI entry point in `installer/bin/ralph.js`, wired in Story 1.2) decides the exit code and whether to render the checklist. This module only provides the data.

## Edge Cases & Gotchas

### 1. **Git Bash on Windows**
   - A user with Git Bash installed may report `bash` as present even on Windows
   - **Handling:** WSL2 detection only blocks if bash is *entirely missing*; Git Bash is sufficient to run the loop (bash is available)
   - **Test:** Mock Windows + bash found → should pass, not fail

### 2. **Version string parsing**
   - Node.js uses "vX.Y.Z" format; some tools use "X.Y.Z"
   - **Handling:** Always strip the leading "v" before parsing
   - **Test:** Mock `process.version = "v20.11.0"` and `"v18.0.0"`

### 3. **TTY detection in CI**
   - CI environments (GitHub Actions, etc.) often have `process.stdout.isTTY = false`
   - **Handling:** The checklist correctly switches to plain text in non-TTY
   - **Test:** Explicitly mock `process.stdout.isTTY = false` and verify no ANSI codes

### 4. **Windows command detection edge case**
   - `where` is inconsistent on some Windows versions; `which` may be available in WSL or Git Bash
   - **Handling:** Use `where` on Windows, `which` on Unix; both return exit code 1 on not found (which we catch)
   - **Test:** Mock both `where` and `which` calls on Windows

### 5. **Slow command lookups**
   - On systems with very long `PATH` variables, `which` / `where` may be slow
   - **Handling:** Preflight is called once at the start of the wizard; ~100ms is acceptable
   - **Test:** No specific test needed; just document in code comments

### 6. **Command availability after install**
   - A user may have git/jq not installed *before* the wizard; the wizard will guide them to install mid-flow
   - **Handling:** Preflight runs once at startup; re-running preflight after install is a future Story 3.2 (update) concern
   - **Test:** No specific test needed; Story 3.2 will add re-check logic if needed
