## User story

As an installer user who has customized my Ralph Loop install, when I re-run the installer to update to a new version, I want my modifications to be preserved safely ŌĆö installer-owned files replaced, user-owned files untouched, and any local changes to installer files presented to me for resolution.

## Acceptance criteria

1. **Update detection:** Running `npx <package> install` (or `update`) in a directory containing `.ralph/manifest.json` enters update mode, showing "installed vX ŌåÆ available vY".

2. **File classification honored:** Installer-owned files from the manifest are replaced; user-owned files are never touched, even if a newer version is available.

3. **Conflict resolution UI:** When an installer-owned file has been locally modified (checksum mismatch), the user is prompted with three options:
   - Keep my changes (skip update for this file)
   - Take the new version (overwrite my changes)
   - Backup and take the new version (rename current to `.backup` and write new)
   
   This prompt respects non-interactive mode: `--update-conflicts` flag controls behavior (valid values: `keep`, `take`, `backup`; invalid values fail non-zero).

4. **User modifications preserved byte-identically:** E2E test: modify `docs/prd.md` (user-owned) and `scripts/prompts/common/project-conventions.md` (installer-owned, assume NOT modified), run update in non-interactive mode ŌåÆ both files survive with zero byte changes.

5. **Backup hygiene:** `.backup` files are written only when the user chooses "backup" for that file; no automatic backups are created on conflicts.

6. **Non-interactive consistency:** `--yes` mode uses a sensible default for conflicts (recommend: `keep`); `--force` skips all prompts and uses the default; invalid `--update-conflicts` values cause non-zero exit before any writes.

7. **Manifest update:** After update completes, `.ralph/manifest.json` is rewritten with:
   - Updated `version` field matching the installer package version
   - New checksums for all replaced files
   - Ownership classes preserved as-is
   - Wizard answers preserved as-is

#### Step 3: Create the update detection and planning module

**File:** `installer/src/updateDetector.js` (new)

This module is responsible for:
1. Loading the manifest from `.ralph/manifest.json`
2. Reading the current installer version from `package.json`
3. Comparing versions and emitting a version-change message
4. Computing a delta: which files are installer-owned and which have been modified locally

**Function signature:**

```javascript
async function detectUpdate(targetDir) {
  // Returns:
  // {
  //   isUpdate: boolean,
  //   installedVersion: string,
  //   availableVersion: string,
  //   manifest: object,
  //   delta: {
  //     installerOwned: [
  //       { path: "scripts/ralph-loop.sh", checksum: "abc123", currentChecksum: "abc123", isModified: false },
  //       { path: "scripts/prompts/...", checksum: "def456", currentChecksum: "xyz789", isModified: true },
  //       ...
  //     ],
  //     userOwned: [
  //       { path: "docs/prd.md", checksum: "ghi789", currentChecksum: "ghi789" },
  //       ...
  //     ],
  //     missing: [ "scripts/prompts/old-prompt.md" ] // was in manifest, no longer exists
  //   }
  // }
}
```

**Key details:**
- Load manifest from `targetDir/.ralph/manifest.json`; if it doesn't exist or is malformed, return `{ isUpdate: false }`
- Read the installer package version from `installer/package.json` (use Node's `require()`)
- For each file in the manifest, compute the current checksum using the same algorithm as the writer (likely `crypto.createHash('sha256')`)
- Mark a file as `isModified` if its current checksum differs from the manifest's checksum
- Separate files by ownership class
- Return the delta for decision-making in the next step

**Tests:**
- Unit test with a fixture manifest: `isUpdate: true`, version mismatch detected
- Unit test with no manifest present: `isUpdate: false`
- Unit test with a modified installer-owned file: `isModified: true` in the delta
- Unit test with a missing file: included in the `delta.missing` list

#### Step 4: Create the conflict resolution prompt module

**File:** `installer/src/updateConflictResolver.js` (new)

When an installer-owned file is modified locally, present the user with three options. This module handles the interactive prompts and non-interactive flag interpretation.

**Function signature:**

```javascript
async function resolveConflicts(conflictFiles, options) {
  // conflictFiles: array of { path, installedChecksum, currentChecksum }
  // options: { yes: boolean, force: boolean, updateConflicts: "keep"|"take"|"backup"|null }
  // Returns: { decisions: { [path]: "keep"|"take"|"backup" }, succeeded: boolean, errors: [] }
}
```

**Behavior:**

- **Non-TTY or `--yes` mode:** Use the default ("keep" is recommended per the requirements)
- **`--force` mode:** Apply the default without prompting; if `--update-conflicts` is set, use that instead
- **Interactive mode:** For each conflicted file:
  1. Display the file path and a brief summary ("modified by you, new version available")
  2. Offer three choices: Keep, Take New, Backup and Take
  3. Record the user's choice
- **Invalid `--update-conflicts` value:** Return `{ succeeded: false, errors: ["Invalid --update-conflicts value: ..."] }`

**Non-interactive flags:**
- `--update-conflicts=keep` ŌĆö keep all locally modified files
- `--update-conflicts=take` ŌĆö take all new versions
- `--update-conflicts=backup` ŌĆö backup all modified files, then take new versions

**Output format (plain text, non-TTY):** One decision per line: `scripts/ralph-loop.sh: keep` or `scripts/prompts/foo.md: backup`

**Tests:**
- Unit test: no conflicts ŌåÆ zero prompts, returns empty decisions
- Unit test: one conflict, interactive ŌåÆ prompt shown, decision recorded
- Unit test: `--yes` flag with conflict ŌåÆ uses default ("keep")
- Unit test: `--force --update-conflicts=take` ŌåÆ applies "take" without prompting
- Unit test: `--update-conflicts=invalid` ŌåÆ returns error, no writes occur

#### Step 5: Extend the writer module to handle update writes

**File:** `installer/src/writer.js` (already exists; extend it)

The writer (Story 2.3) currently handles initial install writes. Extend it to handle updates:

1. **New function:** `async function executeUpdate(targetDir, installPlan, delta, conflictDecisions)`
   - For each installer-owned file in the delta:
     - If `conflictDecisions[path] === "keep"`: skip writing this file
     - If `conflictDecisions[path] === "take"` or `undefined`: write the new file
     - If `conflictDecisions[path] === "backup"`: rename the current file to `<path>.backup`, then write the new file
   - User-owned files are never written (enforced as a pre-condition check)
   - After all writes, call `writeManifest()` to update checksums and version

2. **Manifest rewrite logic:**
   - Load the existing manifest
   - Update the `version` field to match the installer package version
   - Update checksums for all installer-owned files that were written (recompute by reading the files)
   - Preserve `ownership`, `wizardAnswers`, and other metadata as-is
   - Write the updated manifest to `.ralph/manifest.json`

**Pre-condition validation:**
- Verify that the update is legal before writing: no user-owned files are in the write set (enforce as an assertion)

**Tests:**
- Unit test: update with no conflicts ŌåÆ installer-owned files replaced, user-owned untouched, manifest updated
- Unit test: update with conflicts, "keep" decisions ŌåÆ only non-conflicted installer files replaced
- Unit test: update with conflicts, "backup" decisions ŌåÆ conflicted files backed up, new versions written
- Unit test: user-owned files asserted as non-writable (fail if attempted)

#### Step 6: Wire update mode into the CLI

**File:** `installer/src/cli-parser.js` (already exists; extend it) or `installer/bin/ralph.js`

Modify the `install` / `update` command handler to:

1. Classify the target directory
2. If classification is `existing-install`:
   - Call `updateDetector.detectUpdate(targetDir)` to get the delta
   - If `isUpdate === true`:
     - Log the version change message: `"installed v{installedVersion} ŌåÆ available v{availableVersion}"`
     - Summarize the delta: "X installer-owned files to update, Y conflicts, Z user-owned files preserved"
     - Call `updateConflictResolver.resolveConflicts()` with the conflicts and CLI options
     - If resolution fails, exit non-zero with error messages
     - Call `writer.executeUpdate()` with the conflict decisions
   - If `isUpdate === false`: log "Already up to date" and exit 0

3. If classification is `existing-project` or `empty`:
   - Proceed with normal install flow (as before)

**Flag handling:**
- Add `--update-conflicts` flag to the CLI parser
- Pass it through to `resolveConflicts()` in the options object

**Tests:**
- E2E: run installer on an existing install ŌåÆ update mode triggered
- E2E: existing install with conflicts, `--yes` ŌåÆ default behavior applied
- E2E: existing install with conflicts, `--update-conflicts=backup` ŌåÆ backups created

#### Step 7: Create the E2E test for update safety

**File:** `installer/test/e2e-update.test.js` (new)

Implement the PRD success criterion 3: modify user-owned and installer-owned files, run update, verify safety.

**Test scenario:**

```
1. Create a fixture directory (empty or pre-install with manifest)
2. Create an existing install by:
   a. Run install with `--yes` in non-interactive mode
   b. Record checksums of all files
3. Modify two files:
   a. docs/prd.md (user-owned): add a comment at the end
   b. scripts/prompts/common/project-conventions.md (installer-owned): do NOT modify
4. Run update in non-interactive mode with `--update-conflicts=keep`:
   npm exec --cwd=installer -- node bin/ralph.js install --directory <fixture> --yes --update-conflicts=keep
5. Verify:
   a. docs/prd.md is byte-identical to its modified state (user-owned preserved)
   b. scripts/prompts/common/project-conventions.md is replaced with the new version
   c. Manifest checksums are updated for installer-owned files
   d. Command exits 0
```

**Key assertion:** Use `fs.readFileSync()` and `crypto.createHash()` to verify file contents and checksums byte-for-byte.

**Testing a modified installer-owned file (second E2E):**

```
1. Same setup as above
2. Modify scripts/prompts/common/project-conventions.md slightly (e.g., add a comment)
3. Run update with `--update-conflicts=keep`:
   npm exec --cwd=installer -- node bin/ralph.js install --directory <fixture> --yes --update-conflicts=keep
4. Verify:
   a. The file keeps the user's modifications (byte-identical to the modified version)
   b. A .backup file is NOT created (because we chose "keep", not "backup")

5. Run the same update again with `--update-conflicts=backup`:
   npm exec --cwd=installer -- node bin/ralph.js install --directory <fixture> --yes --update-conflicts=backup
6. Verify:
   a. A scripts/prompts/common/project-conventions.md.backup file was created
   b. scripts/prompts/common/project-conventions.md now contains the new upstream version
   c. Manifest checksums are updated to the new upstream versions
```

### Edge cases and gotchas

1. **Manifest version field missing or malformed:**
   - Treat as "no version" (very old or hand-crafted install)
   - Log a warning: "Old manifest format; treating as v0.0.0"
   - Proceed with update as if version mismatch exists

2. **Files deleted by the user:**
   - If a user-owned file is missing, no problem ŌĆö it's not written
   - If an installer-owned file is missing, log a warning but continue: "Expected file `scripts/ralph-loop.sh` not found; skipping"
   - No error; the update proceeds for other files

3. **Manifest file itself is missing:**
   - If `.ralph/manifest.json` is deleted after initial install, `detectUpdate` returns `{ isUpdate: false }`
   - Proceed with normal install flow, which will overwrite files (same as fresh install)
   - This is acceptable; the user can re-run with `--yes` to proceed

4. **Backup file already exists:**
   - If `scripts/prompts/foo.md.backup` already exists and the user chooses "backup":
     - Overwrite the existing `.backup` file without prompting
     - Log a warning: "Overwriting existing backup: `...`"

5. **Checksum collision or corrupt manifest:**
   - If a manifest checksum is invalid (e.g., corrupted file), the file appears "modified"
   - Treat as a regular conflict and prompt the user
   - This is safe; the user can choose "take" to accept the new version

6. **Update across major version boundaries:**
   - No special logic; update proceeds as normal
   - Manifest wizard answers are preserved, so the user doesn't re-answer the wizard
   - Breaking changes (e.g., new template format) are the installer's responsibility to handle gracefully

7. **Concurrent updates:**
   - No locking; if two terminals run update simultaneously, the last one wins
   - This is acceptable for a single-developer tool
   - Document in error messages if manifest is malformed due to concurrent write

8. **Non-interactive conflict when `--update-conflicts` is missing:**
   - If `--yes` is set but `--update-conflicts` is not, use the default ("keep")
   - Log this clearly: "Using default conflict resolution: keep (use `--update-conflicts` to override)"
