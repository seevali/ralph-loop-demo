## Key implementation decisions

### Five-phase flow, .gitignore handled last among installer-owned files

The spec orders removal as: (1) installer-owned files, (2) user-owned files, (3) .gitignore cleanup, (4) manifest, (5) .ralph/ directory. Placing .gitignore cleanup after user-owned files matches the spec's stated file-removal order and keeps the manifest present until the very end — a partial failure leaves evidence for a clean re-run.

### cleanGitignore exported as a pure function

`cleanGitignore(gitignorePath)` is exported so it can be tested directly. It works by splitting the file on newlines, tracking whether it is inside a `# Ralph Loop` block, and excluding lines from that block until a blank line or another comment header. If the resulting content is empty, the file is deleted; otherwise it is rewritten.

### ENOENT treated as success in removeFile

`removeFile` returns `{ success: true }` for ENOENT (file already gone). The goal is "file does not exist"; if that's already true there is nothing to do and no error to report.

### --force suppresses non-critical errors in the return value

With `--force`, `success: true` is returned even if some files could not be deleted (e.g., permission denied). This matches AC7 and the spec's error-handling rule: "force overrides non-critical errors." The manifest itself is still a critical error — its removal failure returns `success: false` regardless of `--force`.

### Injection pattern for testability (no mocking framework)

The `uninstall(options, opts)` signature accepts injectable `log`, `errLog`, `prompts`, `isTTY`, and `noColor` overrides. This lets tests capture output and mock interactive prompts without needing a mocking framework or ESM import patching, consistent with the pattern used throughout the installer (see `writer.js`, `updateConflictResolver.js`).

### Permission test guards against root

Test 7 (partial failure with --force) uses `fs.chmod(scriptDir, 0o555)` to simulate permission denied. A guard `if (typeof process.getuid === 'function' && process.getuid() === 0) return;` skips the test when running as root, where chmod has no effect on file deletion.

## Post-review fixes (2026-06-13)

**Review finding:** `npm test` was flaky (intermittent exit 1, ~1 in 4–6 runs) due to 8 test calls in `uninstall.test.js` that reached the real `console.log`/`console.error` path. When `node --test` runs multiple test files in parallel, it streams results over a V8-serialized binary protocol on each child's stdout. Stray `console.log` output from these tests interleaved with the binary protocol data, intermittently corrupting it ("Unable to deserialize cloned data").

**Fix:** Added `log: () => {}` and `errLog: () => {}` to the `opts` argument of 8 `uninstall()` calls that were exercising success paths without silent log injection:
- Line 259: basic AC1 test (isTTY: false only, no log)
- Line 348: AC4 --yes test (had prompts injection but no log)
- Line 380: AC5 --force test (had prompts injection but no log)
- Line 409: AC5 .ralph/ removal test (isTTY: false only)
- Line 434: AC6 .gitignore cleanup test (isTTY: false only)
- Line 459: AC6 .gitignore delete test (isTTY: false only)
- Line 490: AC7 partial failure --force test (had errLog injection but no log)
- Line 591: no user-owned files test (had prompts injection but no log)

After fix: 10 consecutive `npm test` runs all exit 0 with 219 pass, 0 fail. No stray uninstall log output in test stdout.
