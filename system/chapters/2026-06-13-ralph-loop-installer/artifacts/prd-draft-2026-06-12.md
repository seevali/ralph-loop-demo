## 1. Goals and Background Context

### 1.1 Goals

- Allow any developer to install the Ralph Loop into **an empty directory or an existing project folder** with a single command and minimal effort.
- Make the installation experience feel as polished and approachable as the BMAD Method installer (`npx bmad-method install`), which is the explicit benchmark for this project.
- Lower the barrier for **average and early AI adopters** who currently find the Ralph Loop hard to grasp — the installer should teach the concepts as it installs.
- Make updates as easy as the first install (re-run the same command), without destroying user customizations.
- Keep the installer itself simple enough that it does not become a bigger maintenance burden than the loop it installs.

### 1.2 Background

The Ralph Loop (after Geoffrey Huntley's "Ralph Wiggum" technique) runs an AI coding agent (Claude Code by default) repeatedly against a prompt and a task/PRD file until completion criteria are met. The open-sourced demo at `\demos\ralph-loop-demo` works, but feedback from early users is consistent: the value is clear, the setup is not. Users must currently clone the repo, understand its file layout, copy files into their own project by hand, edit configuration, and understand loop concepts (prompt file, PRD/task file, progress/memory files, completion signal, iteration limits) before anything runs.

The BMAD Method solved the identical adoption problem with a guided npx installer: one command, a short interactive wizard, sensible defaults, non-interactive flags for automation, and an update path that preserves customizations. This PRD specifies an equivalent installer for the Ralph Loop.

### 1.3 Competitive / prior-art research summary

**BMAD Method installer (benchmark)**
- Single entry point: `npx bmad-method install`. The same command performs first install, upgrade, and channel switching.
- Prerequisite check: Node.js v20+ enforced at install time.
- Interactive wizard asks: target directory, which modules to install, which AI tools/IDEs to integrate with (Claude Code, Cursor, etc.).
- Non-interactive mode for CI/CD: `npx bmad-method install --directory /path --modules bmm --tools claude-code --yes`, with `--set <module>.<key>=<value>` overrides and `--list-options` discovery.
- Release channels: `@stable` (default) and `@next` (prerelease).
- Updates preserve user customizations.
- Installs into a dedicated folder (`_bmad/`) inside the user's project, keeping its footprint contained.
- After install, the user is told exactly what to do next ("open your AI IDE in this folder", "run /bmad-help").

**Other public Ralph Loop installers (gap analysis)**
- `PageAI-Pro/ralph-loop`: installed by running a command in the project directory; supports empty-directory starts but discourages them; ships a `prd-creator` skill; relies on Docker Sandboxes. No guided wizard.
- `snarktank/ralph`: manual copy of `prompt.md` / `CLAUDE.md` plus `prd.json` into the project; memory via git history, `progress.txt`, `prd.json`; completion via a `<promise>COMPLETE</promise>` signal. No installer at all.
- `frankbria/ralph-claude-code`: installs a global `ralph` command usable in any directory; adds exit detection, rate limiting, session management. Global install model rather than per-project scaffold.
- `syuya2036/ralph-loop`: agent-agnostic shell script; user passes the agent CLI command as an argument. Manual setup.

**Conclusion:** no existing Ralph implementation offers a BMAD-quality guided installation. A polished `npx` wizard is a genuine differentiator, not a parity feature.

**Recommended CLI technology (research outcome)**
- **`@clack/prompts`** for the interactive wizard: the current standard for scaffolding CLIs (used by create-astro and similar tools). Provides intro/outro framing, text/select/multiselect/confirm prompts, spinners, grouped multi-step wizards, and built-in cancel handling, with a ~2KB footprint that keeps `npx` cold starts fast. Ink (React-based) is rejected for this use case due to bundle size and cold-start cost; Inquirer.js is rejected as the legacy option with weaker wizard-flow ergonomics.
- **Commander.js** for argument/flag parsing and subcommands, with `parseAsync`, standardized exit codes, and graceful SIGINT handling.
- **Node.js ≥ 20** as the only runtime prerequisite (matches BMAD), distributed via npm and invoked with `npx` so nothing is permanently installed.
- Supporting libraries kept minimal: `picocolors` (or clack's built-ins) for color, no chalk/ora/figlet.

## 2. Assumptions and Constraints

> **IMPORTANT — repo discovery required.** This PRD was written without direct access to the current contents of `\demos\ralph-loop-demo`. The assumptions below describe the *typical* Ralph Loop layout. Epic 1, Story 1.1 requires the implementing agent to inventory the actual repo and reconcile any differences before building. Where the real repo differs, the real repo wins and this PRD's file names should be mapped accordingly.
>
> **[Reconciliation done 2026-06-13 — see the chapter plan. Every assumption in §2.1 below was wrong in at least one particular; the table in the plan's "Reconciliation" section is binding.]**

### 2.1 Assumed current state of the Ralph Loop demo

- A loop runner script (bash, e.g. `ralph.sh` or `loop.sh`) that invokes an agent CLI (Claude Code by default) in a loop.
- A prompt file (`PROMPT.md` or `CLAUDE.md` guidance) defining per-iteration instructions.
- A task/spec source the loop works through (e.g. `prd.json`, `fix_plan.md`, or `specs/` directory).
- Memory/progress artifacts (e.g. `progress.txt`, learnings file, git history conventions).
- A completion-detection mechanism (completion signal string, all-tasks-done check, or max-iteration cap).
- A README explaining the above — currently the only onboarding path.

### 2.2 Constraints

- C1: The installer must not require Docker, Python, or anything beyond Node.js ≥ 20 and git.
- C2: The installed loop must keep working in the environments the demo already supports (at minimum WSL2/Linux/macOS bash; see NFR-3 for Windows).
- C3: The installer is distributed under the same open-source license as the demo repo.
- C4: The implementation itself will be carried out by the Ralph Loop using this PRD (dogfooding), after BMAD planning produces stories — so requirements must be unambiguous and independently verifiable.

## 3. Users and Personas

- **P1 — Early AI adopter ("the cousin's audience", primary).** Knows what Claude Code is, has run a few prompts, has never run an agent loop. Needs hand-holding, explanation, and safe defaults. Success = loop running on a toy project within 5 minutes without reading the README.
- **P2 — Experienced agentic developer.** Already runs Claude Code daily, maybe used BMAD. Wants speed: skim the wizard, accept defaults, or skip it entirely with flags. Success = installed and customized in under a minute.
- **P3 — Automation / CI user.** Wants fully non-interactive install in scripts, containers, or as part of a bigger toolchain. Success = one deterministic command, exit code 0, no prompts.
